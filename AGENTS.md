## 1. 基本方針

- 返答、説明、作業メモは日本語で行う。
- ユーザーは非エンジニアまたは初学者の可能性がある前提で進める。
- 専門用語を使う場合は、短い補足説明を添える。
- 曖昧な表現を避け、何をしたか・何をしていないかを明確に書く。
- 可能な限り、ユーザーが次に迷わない形で結果を残す。

## 2. 実行環境の原則

- スクリプトやコマンドの実行は、可能な限りローカル環境を優先する。
- サンドボックスや隔離環境での実行は、必要な場合に限る。
- サンドボックスや例外的な実行環境を使う場合は、開始前に短く報告する。
- 実行環境の制約でローカル実行が難しい場合は、その理由を明記する。
- 実行場所の選択は、速度だけでなく、安全性、再現性、保守性を優先して判断する。
- 実行後は、結果だけでなく、どこで何を実行したかも分かるように残す。

## 3. ファイル管理と変更方針

- いきなり変更せず、まず対象ファイル、関連ファイル、実行フローを確認する。
- 変更は必要最小限にとどめ、関係ない箇所は触らない。
- 不要なコピー、重複ファイル、無目的な一時ファイルは増やさない。
- 既存の命名規則、フォルダ構成、文体、コメント方針があれば合わせる。
- 不明点があっても、安全に進められる範囲は自律的に進める。
- 破壊的な操作、大量削除、全面置換は慎重に扱い、必要性を明確にする。

## 4. 作業フロー

1. 必要なファイルや実行対象を確認する。
2. スクリプトや設定変更を作成・更新する。
3. 適切な環境で実行する。
4. 結果を確認し、必要なら修正して再実行する。
5. 完了内容と未解決事項を記録する。

## 4.1 Git 運用ルール

- Git の運用は、原則として AI が自動で管理する。
- 作業がひと区切りついたら、以下を自律的に確認する。
  - `git status`
  - 差分内容
  - コミット要否
  - Push 要否
- コミットは、あとで履歴を見返したときに意味が分かる単位で分ける。
- `force push`、履歴破壊、不要な `amend` はしない。
- 関係ない変更や、ユーザーが作った未整理変更は巻き込まない。
- タグ付けや本番反映の節目は影響が大きいため、必要に応じてユーザーへ共有してから進める。

## 5. 説明と報告のルール

- 実施内容は「何を目的に」「何を変更したか」「何が未完了か」が分かるように報告する。
- コードや設定を変えた場合は、影響範囲を短く添える。
- 調査のみで変更ファイルがない場合でも、確認した内容と判断結果を記録する。
- 難しい内容は、先に結論、その後に短い補足を書く。

## 6. 記録ファイル運用

- 新しく作る Markdown ファイルや更新する Markdown ファイルの `更新日:` 表記には、日付だけでなく時刻まで入れる。
- 形式は `YYYY-MM-DD HH:MM JST` にそろえる。

### 秘書ノート

- Obsidian 側の入口メモは `/Users/marupro/Library/Mobile Documents/iCloud~md~obsidian/Documents/AFROG電脳/10_デジタルスキル/00_PROJECT/FX/トレード支援システム/👩‍⚖️秘書.md` を使う。
- `👩‍⚖️秘書.md` は履歴置き場ではなく、常に最新版だけを残す入口メモとして使う。
- 毎回追記ではなく、必要に応じて全体を書き換える。
- 毎作業後に必ず更新する。
- 基本構成は次の 4 ブロックに固定する。
  - `## 今の状況`
  - `## 次にやる`
  - `## 次に見る`
  - `## 重要メモ`
- `今の状況` は 1〜3 行の短い要約にとどめる。
- `次にやる` は 1〜3 件までに絞る。
- `次に見る` は今すぐ開くべき資料リンクを 2〜5 件に絞る。
- `重要メモ` は今の判断に必要な前提だけを 1〜3 件書く。
- 文体は、変数名や機械的な列挙を並べるより、人が次の人へ短く報告する言い方を優先する。
- `今の状況` と `次にやる` は、専門用語や値をそのまま並べるだけで終わらせず、「いま何待ちか」「次に何を確かめるか」が自然に伝わる短文にする。
- 大きな報告や長い背景説明は `👩‍⚖️秘書.md` に持ち込まず、`📒打ち合わせノート.md` か `運用資料/開発ロードマップ.md` 側へ逃がす。
- `👩‍⚖️秘書.md` は「どこに何があるか」の一覧ではなく、「今どういう状態で、次に何をするか」が先に分かる入口にする。

### スレッド引き継ぎファイル

- `運用資料/スレッド引き継ぎファイル.md` は、次スレッドで実務を再開するための圧縮引き継ぎに使う。
- 日々の進捗を毎回追記する場所ではなく、スレッド切り替え直前や大きな区切りで精査して更新する。
- 役割は、次スレッドで最初に必要な前提、最優先タスク、ブロッカー、重要ルールを短く渡すこと。
- 長い経緯は `progress.md` に残し、引き継ぎファイルには圧縮した現在地だけを書く。
- 新スレッドでは、まず `👩‍⚖️秘書.md` → `運用資料/開発ロードマップ.md` → `運用資料/NEXT_TASK.md` までを読み、そこで足りない場合だけ `運用資料/スレッド引き継ぎファイル.md` を開く。

### progress.md

- 作業終了時に必ず更新する。
- 追記型で運用し、新しい記録をファイルの上に追加する。
- 毎回、以下を簡潔な箇条書きで書く。
  - 日時
  - 実施内容
  - 変更ファイル
  - 未解決事項
  - メモ
- `日時` は `YYYY-MM-DD HH:MM JST` 形式で記録する。
- `日時` は推測ではなく、更新直前に `date` コマンドなどで取得した実時刻を使う。
- 変更ファイルがない場合でも、調査・検討した内容を残す。
- 未完了項目がある場合は、曖昧にせず明記する。

### NEXT_TASK.md

- 作業終了時に必ず更新する。
- 常に最新版だけを残し、古い候補を蓄積しない。
- 基本構成は `現在の状況`、`次のタスク`、`ブロッカー`、`完了条件` を推奨する。
- `現在の状況` は履歴ではなく、次回判断に必要な前提だけを残す整理欄として扱う。
- `現在の状況` は、不要になった項目を消す、似た内容をまとめる、優先度順に並べ替える、表現を短く整える、という更新をしてよい。
- 次回やることを優先順で 3〜7 件書く。
- 最初の 1 件は、すぐ着手できる具体的な作業にする。
- ブロッカーがあれば明記する。
- 完了条件がある場合は 1〜2 行で書く。

## 7. 作業終了時の必須出力

## 8. コード変更時のルール

- 変更理由を説明できない修正はしない。
- 一時しのぎの値の直書きは避け、設定ファイルや定数化を優先する。
- 既存処理を壊す可能性がある場合は、そのリスクを明記する。
- テストや確認方法がある場合は、実施したか未実施かを報告する。

## 9. 例外的な環境を使う場合

以下のような場合は、例外的な実行環境の利用を検討し、事前に報告する。

- 実験的または破壊的な処理で、ローカル環境に影響を与える可能性がある場合
- ローカルに必要な実行環境や依存関係が不足している場合
- ローカルでは再現しにくい隔離検証が必要な場合
- 大量処理や検証分離のために、安全な別環境が適している場合
