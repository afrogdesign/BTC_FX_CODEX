# ログ検証と改善運用ガイド

このファイルは、BTC監視システムのログを使って「何を確認し、どう改善するか」を定型化するための運用ガイドです。

補足:
- 「どの環境を正本にするか」「Ver01 / Ver02 をどう並走させるか」は `今後の運用ルール.md` を基準にしてください。
- このファイルは、主にログ確認と改善判断の流れに絞ったガイドです。

## 1. 目的
- 監視ロジックの品質を、感覚ではなくログで評価する。
- 通知の質（多すぎる/少なすぎる）を改善する。
- 月次・年次で傾向を比較し、設定値の見直し根拠を作る。

## 2. まず見るファイル
- `logs/last_result.json`: 直近1回の判定詳細
- `logs/csv/trades.csv`: 連続実行の時系列ログ（集計の主対象）
- `logs/csv/signal_outcomes.csv`: 通知後の事後評価ログ
- `logs/csv/user_reviews.csv`: Obsidian から取り込んだユーザーレビュー
- `logs/csv/shadow_log.csv`: 観測・事後評価・レビューを統合した改善判断用マスターログ
- `logs/signals/*.json`: 各サイクルの詳細スナップショット
- `logs/errors/*.log`: 例外ログ

## 4. レポートの見方
`tools/log_analytics.py` が出力する項目:
- 全体KPI
  - 平均 `confidence`
  - `primary_setup_status=invalid` 比率
  - `ai_error` 発生率
- 分布
  - `bias`, `phase`, `market_regime`, `primary_setup_status`
- `no_trade_flags` 上位
  - 何がボトルネックかを把握
- 年次集計 / 月次集計
  - 件数、平均confidence、invalid率、ai_error率
- 時間帯別（JST）
  - どの時間帯で品質低下が起きやすいか

`tools/log_feedback.py` が出力する項目:
- `shadow_log.csv` 基準の統合集計
  - `regime`, `prelabel`, `signal_tier`, `bias`, `data_quality_flag`
- signal-based / entry-ready-based の MFE / MAE
  - 方向感が良かったか、ready で入っていたらどうだったかを分けて確認
- 通知監査
  - A: 通知して良かった
  - B: 通知したが微妙
  - C: 通知しなかったが本当は良かった
  - D: 通知しなかったので正解
- ユーザーレビュー集計
  - `too_early`, `low_value`, `actual_move_driver` 入力率
- 改善候補
  - その週・月で直す価値が高い上位候補

## 5. 改善に直結する見方
### 5-1. invalid率が高いとき
- `RR_insufficient` が多いか確認。
- 多い場合は以下を優先検証:
  - `MIN_RR_RATIO`
  - S/R検出ロジック（ゾーン幅）
  - `SL_ATR_MULTIPLIER`

### 5-2. ai_error率が高いとき
- `AI_TIMEOUT_SEC` が短すぎないか確認。
- `AI_RETRY_COUNT` を増やす余地を確認。
- 回線・API障害時のフォールバック運用を見直し。

### 5-3. 偏りが強いとき（例: shortばかり）
- `market_regime` と一致しているかを確認。
- 一致していない場合、スコア閾値または構造判定ロジックを再評価。

## 6. 定例運用（おすすめ）
- 毎日: まず Ver02 の `heartbeat.txt`、`last_result.json`、`launchctl` の `running / pid` を確認
- 毎日: 次に Ver01 / Ver02 の通知有無、`confidence`、suppress 理由、`notify_reason_codes` を見比べる
- 毎日: `Ver02.1 API` / `Ver02.1 CLI` の差分観測がある日は `運用資料/reports/cli_api定期比較レポート.md` を更新し、CLI の安定性を同じ観点で追う
- 毎日: 通知から24時間経過した対象がある場合だけ `tools/log_feedback.py daily-sync` を実行
- 毎週: 全期間レポートと週次フィードバックレポートを更新して改善候補を3件までに絞る
- 毎月: 月次レポートと月次フィードバックレポートで設定変更の是非を判断
- 毎年: 年次レポートで運用方針を見直す
