# 実行系反映・再起動・報告手順

更新日: 2026-05-26 JST

## 目的

コード変更のうち、常駐 `com.afrog.btc-monitor` や日次レポート実行系に影響するものがある場合に、

1. 実装
2. テスト
3. 必要なレポート生成
4. 実行系の再起動
5. 実行状態確認
6. repo 内の正本更新
7. Global_BOX 更新
8. 人向け報告
9. commit / push

までを 1 つの作業単位として漏れなく閉じるための手順。

## この手順を使うトリガー文言

ユーザーが次のように言った場合、Codex はこの手順を一連で実行する。

```text
実行系まで反映して
```

同義として扱ってよい例:

- `再起動までやって`
- `実行ファイルまで最新化して`
- `runtime 反映までお願い`

ただし、明らかに再起動不要な文書修正だけの依頼には機械的に適用しない。

## 再起動対象に含める変更

以下のいずれかを触った場合は、再起動要否を確認する。

- `main.py`
- `src/` 配下の監視本体ロジック
- `config.py`
- `src/analysis/`、`src/trade/`、`src/ai/` の実行時参照コード
- `tools/run_daily_reports.py` のような運用実行ファイル
- `deploy/com.afrog.btc-monitor.plist`
- `tools/start_monitor.sh`

## 一連の作業

### 1. 実装と検証を閉じる

- 実装を完了する
- 既存の主要テストを通す
- 必要な追加テストを通す
- `git diff --check` を確認する

### 2. 必要な生成物を更新する

- 必要なら `daily-sync`
- 必要なら分析レポート
- 必要なら `report_hub_latest.md`

`tools/run_daily_reports.py` でまとめて回せるなら、それを優先してよい。

### 3. 実行系を再起動する

監視本体に影響する変更がある場合は:

```bash
cd /Users/marupro/CODEX/01_active/BTC_FX_CODEX/btc_monitor
zsh tools/start_monitor.sh
```

日次レポート常駐を将来導入した後にその対象変更がある場合は、そのジョブも同様に扱う。
ただし 2026-05-26 JST 時点では `run_daily_reports.py` は launchd 常駐化していないため、例示ファイル追加だけでは再起動対象にしない。

### 4. 実行状態を確認する

最低限、次を確認する。

```bash
launchctl print "gui/$(id -u)/com.afrog.btc-monitor"
ls -lT logs/runtime/monitor.err logs/heartbeat.txt logs/last_result.json
cat logs/runtime/monitor.pid
```

確認項目:

- `state=running`
- `pid` が取れる
- `working directory` が repo を向いている
- `monitor.err` が空、または許容済み warning のみ
- `heartbeat` / `last_result` が次サイクルで更新される見込みである

### 5. repo 内の正本を更新する

最低限更新する:

- `運用資料/NEXT_TASK.md`
- `運用資料/履歴/progress.md`

必要なら更新する:

- `運用資料/運用/実務/運用コマンドメモ.md`
- `運用資料/reports/report_hub_latest.md`

記載する内容:

- 実行ブランチ
- 再起動した対象
- 再起動後 PID
- エラー有無
- 関連レポート更新有無
- 次に見る確認項目

### 6. Global_BOX を更新する

デプロイ先の実態や実行コマンド、常駐構成、実行 branch の扱いが変わる場合は更新する。

主な対象:

- `00_Global_BOX/20_環境情報/デプロイ先/一覧.md`
- `00_Global_BOX/20_環境情報/デプロイ先/ラボ iMac.md`

原則:

- branch 名は `NEXT_TASK.md` を正本とする
- 件名ラベル版名と branch 名を混同しない
- 実態が変わっていない箇所は触らない

### 7. 人向け報告を更新する

更新対象:

- `👩‍⚖️秘書.md`
- `📒打ち合わせノート.md`

少なくとも次を書く:

- 最新 commit
- 再起動済みか
- PID / state
- 何を反映したか
- 次に見る点

### 8. commit / push する

- repo 内変更を commit
- push
- `git status` が空であることを確認する

### 9. 仕様書運用を閉じる

`chatgpt/specs/active/` に対応する作業なら、

- 実施済み spec を `archive/` へ移す
- `active/` を未着手だけの状態にする

## 完了条件

- 必要なテストが通っている
- 必要なレポートが更新されている
- 実行系変更がある場合、`com.afrog.btc-monitor` が再起動済み
- `launchctl` / PID / error の確認が済んでいる
- `NEXT_TASK.md` と `progress.md` が更新されている
- 実態変更がある場合は Global_BOX が更新されている
- 人向け報告が更新されている
- commit / push 後に作業木がクリーン
