# ログ検証と改善運用ガイド

更新日: 2026-03-23 18:00 JST

ログ改善で迷ったときの最短ガイドです。

## まず見る

- `logs/last_result.json`
- `logs/csv/trades.csv`
- `logs/csv/signal_outcomes.csv`
- `logs/csv/user_reviews.csv`
- `logs/csv/shadow_log.csv`
- `logs/errors/*.log`

## 日次でやること

1. `tmp/status/prod_status_summary.md` で本番の生存確認
2. 通知や異常が出た回だけ `last_result.json` と `signals/*.json` を確認
3. 通知から 24 時間経過した対象があるときだけ `daily-sync`

## `daily-sync`

開発:

```bash
./.venv312/bin/python tools/log_feedback.py daily-sync
```

本番:

```bash
./.venv312_prod/bin/python tools/log_feedback.py daily-sync
```

## 改善判断の見方

- 通知が少なすぎる
  - `confidence`
  - `suppress_reason_codes`
  - `RR_insufficient`
- 通知品質が悪い
  - `notify_reason_codes`
  - `user_reviews.csv`
  - `shadow_log.csv`
- AI 周りが不安定
  - `ai_advice_error`
  - `ai_summary_error`
  - `system_mode_label`

## 改善時の原則

- 単発の印象で閾値を変えない
- まず `trades.csv` と `signal_outcomes.csv` の傾向を見る
- 本文改善は AI 自由作文より、コード側テンプレート主導を優先する
- 重い分析レポートは必要なときだけ作る
