# BTC Monitor 今後の運用ルール

更新日: 2026-05-18 JST

## 基本方針

- 凍結済みの本番履歴は `MBP2020` にある
- 現行の主作業は `iMac 2019` で進める
- 実運用の判断は本番ログと `launchd` 状態を優先する
- Mac 横断のデプロイ先一覧は `Global_BOX` を見る

## 環境の役割

- `MBP2020`
  - 停止済み `Ver02.1` の archive 参照先
- `MBA M4`
  - 持ち運び前提の作業機。常設補助ジョブは置かない
- `iMac 2019`
  - 現行 `Ver02.5-v8` の観測と確認の主作業機
  - `daily-sync` と `sync-ai-post-reviews` の運用確認先

## 2台並行作業のルール

- `iMac 2019` と `MBA M4` を同日に使ってよいが、同時に別々の未同期変更を溜めない
- 実装とテスト、軽量同期、ログ取得の主作業は `iMac 2019` を優先する
- `MBA M4` は持ち運び時のコード修正、確認、軽い文書修正を優先し、常設ジョブ前提で考えない
- Mac を切り替える前に commit と push を終え、次の Mac では必ず `git pull --rebase` 後に作業する
- 同じファイルを両 Mac で並行編集しない。特に `NEXT_TASK.md`、`履歴/progress.md`、`運用資料/運用/` 配下は片方で確定してから渡す
- 競合した場合は履歴系ファイルを優先して人手で整理し、`ours` / `theirs` で雑に潰さない

## 切替チェック

1. 作業開始前に `git status` が空か確認する
2. 作業開始前に `git pull --rebase` を実行する
3. 区切りがついたら commit する
4. Mac を離れる前に push する
5. 判断変更は `NEXT_TASK.md`、実施事実は `履歴/progress.md` に残す

記録変更で迷ったときは、役割定義を優先し、詳細は `記録ファイル運用ルール.md` を参照する。

## 日常の確認順

1. 開発側 `logs/last_result.json` と `logs/heartbeat.txt`
2. 必要なときだけ本番 `launchctl` と本番 `logs/last_result.json`
3. 異常時だけ `tools/pull_ver021_prod_logs_auto.sh`

## 通知・改善の方針

- 直近は `Phase 0` 本番観測、`Phase 1A` 観測紙トレード、`Phase 1B-lite` 限定検証を優先する
- 本文改善は AI 自由作文より、コード側テンプレート主導を優先する
- `Phase 1B-lite` は正式 `Phase 1B` でも実弾でもないため、専用CSVで分けて見る
- `Ver03` は `Phase 0` 一周完了と `Phase 1` 中核導入がそろうまで名乗らない

## 更新時の原則

- デプロイ先の実態が変わったら `Global_BOX` の `デプロイ先/一覧.md` と該当ページを更新する
- BTC 側には案件固有メモだけ残し、共通一覧の複製は増やさない
