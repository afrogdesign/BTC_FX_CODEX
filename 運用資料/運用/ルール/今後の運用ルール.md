# BTC Monitor 今後の運用ルール

更新日: 2026-03-23 18:35 JST

このファイルは、いまの運用判断だけを残す短い正本です。

## 基本方針

- 本番の正本は `MBP2020`
- 開発は `MBA M4` と `iMac 2019` を使い分ける
- 実運用の判断は本番ログと `launchd` 状態を優先する
- Mac 横断の実行ファイル一覧は `Global_BOX` を正本にする

## 環境の役割

- `MBP2020`
  - 本番常駐 `com.afrog.btc-monitor-ver021`
  - 本番 `.env` は `cli / cli`
  - 実行時は `CLI` 優先、失敗時だけ `API` 自動フォールバック
- `MBA M4`
  - 持ち運び前提の作業機。常設補助ジョブは置かない
- `iMac 2019`
  - 本番軽量同期 `com.afrog.btc-monitor-status-sync` の常設先

## 2台並行作業のルール

- `iMac 2019` と `MBA M4` を同日に使ってよいが、同時に別々の未同期変更を溜めない
- 実装とテスト、軽量同期、ログ取得の主作業は `iMac 2019` を優先する
- `MBA M4` は持ち運び時のコード修正、確認、軽い文書修正を優先し、常設ジョブ前提で考えない
- Mac を切り替える前に commit と push を終え、次の Mac では必ず `git pull --rebase` 後に作業する
- 同じファイルを両 Mac で並行編集しない。特に `NEXT_TASK.md`、`progress.md`、`運用資料/運用/` 配下は片方で確定してから渡す
- 競合した場合は履歴系ファイルを優先して人手で整理し、`ours` / `theirs` で雑に潰さない

## 切替チェック

1. 作業開始前に `git status` が空か確認する
2. 作業開始前に `git pull --rebase` を実行する
3. 区切りがついたら commit する
4. Mac を離れる前に push する
5. 判断変更は `NEXT_TASK.md`、実施事実は `progress.md` に残す

## 日常の確認順

1. `tmp/status/prod_status_summary.md`
2. `tmp/status/prod_status_sync_last_success.txt`
3. 必要なときだけ本番 `launchctl` と `logs/last_result.json`
4. 異常時だけ `tools/pull_ver021_prod_logs_auto.sh`

## 通知・改善の方針

- 直近は `Phase 0` の本番観測を優先する
- 本文改善は AI 自由作文より、コード側テンプレート主導を優先する
- `Phase 1` はレビュー反映と実データ評価の準備段階
- `Ver03` は `Phase 0` 一周完了と `Phase 1` 中核導入がそろうまで名乗らない

## 更新時の原則

- 本番ラベル、実行ファイル、起動方式、主ログが変わったら `Global_BOX` の本番環境ページも更新する
- 開発環境の常駐が変わったら `Global_BOX` の該当 Mac ページも更新する
- BTC 側には案件固有メモだけ残し、共通一覧の複製は増やさない
