# BTC Monitor 今後の運用ルール

このファイルは、Ver02 を本番環境 MBP2020 へ移したあとの運用ルールをまとめたものです。
今後は「どこで動かすか」「何を毎日見るか」「どの条件で次フェーズへ進むか」を、このファイルを基準に判断します。

## 1. 基本方針

- 常駐運用は MBP2020 を正本にする。
  - 正本: いちばん信頼して判断に使う本体データ
- MBA15 は開発・検証専用にする。
- 実運用の事実確認は、MBP2020 上のログと `launchd` 状態を優先する。
- ローカル実行結果は参考情報として扱い、最終判断は本番ログで行う。

## 2. 環境ごとの役割

### 本番環境 MBP2020

- Ver01 を比較基準として残す。
- Ver02 を改善版として並走させる。
- 常駐ジョブは本番機だけで動かす。
- 監視・通知・事後評価ログの継続運用は本番機で行う。

### 開発環境 MBA15

- 常駐プロセスは置かない。
- 使い道は以下に限定する。
  - コード修正
  - 単発確認
  - 手動コマンド実行
  - 一時的な検証
- 検証後は常駐設定や残しっぱなしのプロセスを残さない。

## 3. バージョン分離ルール

- Ver01 と Ver02 は、同じ本番機でも完全分離を維持する。
- 分離対象は以下。
  - 配置ディレクトリ
  - `launchd` ラベル
  - plist
  - `.env`
  - `logs/`
- どちらか一方の変更で、もう一方のログや起動設定を上書きしない。

## 4. 日次運用ルール

### 4-1. まず Ver02 の生存確認

毎日最初に、Ver02 の以下を確認する。

- `logs/heartbeat.txt`
- `logs/last_result.json`
- `launchctl print ...` の `state = running` と `pid = ...`

この3つがそろっていれば、「常駐は生きていて直近判定も更新されている」と判断しやすい。

補足:

- 2026-03-12 10:44 JST 時点では、MBP2020 実機でこの3点を確認済み。
- Ver02 の `heartbeat.txt` は `2026-03-12 10:05:00 JST`、`last_result.json` は `2026-03-12 10:05:24 JST`、`pid` は `98787` だった。

### 4-2. 次に Ver01 / Ver02 の差分確認

比較軸は固定する。

- 通知有無
- `confidence`
- suppress 理由
- `notify_reason_codes`
- 通知の納得感

単発の印象ではなく、同じ軸で見比べる。

補足:

- 2026-03-12 10:44 JST 時点の最新実測では、Ver01 は `bias=wait`、`confidence=30`、Ver02 は `was_notified=False`、`bias=long`、`confidence=0` だった。
- Ver01 の最新 `last_result.json` / `trades.csv` では `signal_id` と `system_label` が空で、Ver02 と記録フォーマット差がある。

### 4-3. 通知発生時の必須確認

通知が発生したら、そのケースは必ずレビュー対象にする。

- `logs/csv/trades.csv`
- `logs/signals/*.json`
- `📝通知レビュー.md`

見るポイント:

- `was_notified=True` が入っているか
- `notify_reason_codes` が保存されているか
- `signals/*.json` に通知理由が残っているか

### 4-4. 24時間後の事後評価

通知から 24 時間経過したら、`daily-sync` を実行する。

```bash
./.venv312_prod/bin/python tools/log_feedback.py daily-sync
```

補足:

- 上のコマンドは、本番 MBP2020 の Ver02 環境で実行する前提です。
- ローカル検証でだけ回す場合は、開発環境側の仮想環境パスに読み替えます。

目的:

- 事後評価ログを確定する
- レビュー候補を更新する
- `done` にしたレビューを CSV に反映する

確認対象:

- `logs/csv/signal_outcomes.csv`
- `logs/csv/shadow_log.csv`
- `logs/csv/user_reviews.csv`
- `📝通知レビュー.md`

### 4-5. 作業記録

その日の確認や変更が終わったら、以下を更新する。

- 進行記録: `運用資料/progress.md`
- 次回作業: `運用資料/NEXT_TASK.md`
- 中長期メモ: `📒打ち合わせノート.md`

## 5. 判断ルール

### 5-1. すぐ不具合扱いしない条件

- Ver02 は「通知が少ない」だけでは不具合扱いしない。
- `suppress_reason_codes` と `confidence` の整合が取れていれば、まず観察継続にする。

### 5-2. 不具合とみなす条件

以下のどれかが出たら、運用確認ではなく不具合調査へ切り替える。

- `heartbeat.txt` が更新されない
- `last_result.json` が止まる
- `trades.csv` に新規行が増えない
- 通知時に `notify_reason_codes` が空
- `daily-sync` 後に `signal_outcomes.csv` / `user_reviews.csv` が更新されない
- `actual_move_driver` を入れても `logic_validated` が反映されない

### 5-3. 閾値調整のルール

- 閾値調整は単発の印象で行わない。
- 少なくとも以下のどちらかを見てから判断する。
  - 数件の通知レビュー
  - 一定期間の抑制ログ

## 6. フェーズ進行ルール

### Phase 0 を継続する条件

- Ver02 常駐が安定している
- 実通知が発生している
- `daily-sync` が本番で一周完了している
- `actual_move_driver` から `logic_validated` まで反映確認が終わっている

### Phase 1 へ進む条件

- 上の未確認事項がすべて解消している
- Ver02 のログ設計に追加修正が不要と判断できる

### Ver01 を止める条件

- Ver01 はすぐ止めない
- まず比較対象として残す
- 停止判断は、以下のような明示条件を満たしてから行う
  - Ver02 が一定期間安定運用できた
  - 通知品質が Ver01 以上と判断できた
