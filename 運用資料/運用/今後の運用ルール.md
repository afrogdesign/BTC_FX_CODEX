# BTC Monitor 今後の運用ルール

このファイルは、Ver02.1 を本番環境 MBP2020 へ置いたあとの運用ルールをまとめたものです。
今後は「どこで動かすか」「何を毎日見るか」「どの条件で次フェーズへ進むか」を、このファイルを基準に判断します。

## 1. 基本方針

- 常駐運用は MBP2020 を正本にする。
  - 正本: いちばん信頼して判断に使う本体データ
- MBA15 は開発・検証専用にする。
- 実運用の事実確認は、MBP2020 上のログと `launchd` 状態を優先する。
- ローカル実行結果は参考情報として扱い、最終判断は本番ログで行う。

## 2. 環境ごとの役割

### 本番環境 MBP2020

- Ver01 を比較基準として残す。
- Ver02.1 を改善版として並走させる。
- 本番は `API`、開発は `CLI` を基本形にする。
- 常駐ジョブは本番機だけで動かす。
- 監視・通知・事後評価ログの継続運用は本番機で行う。

### 開発環境 MBA15

- 常駐プロセスは置かない。
- 使い道は以下に限定する。
  - コード修正
  - 単発確認
  - 手動コマンド実行
  - 一時的な検証
- 検証後は常駐設定や残しっぱなしのプロセスを残さない。

## 3. バージョン分離ルール

- Ver01 と Ver02.1 は、同じ本番機でも完全分離を維持する。
- 分離対象は以下。
  - 配置ディレクトリ
  - `launchd` ラベル
  - plist
  - `.env`
  - `logs/`
- どちらか一方の変更で、もう一方のログや起動設定を上書きしない。

## 3.1 コードと実データの分け方

- Git はコード管理に使う。
- `logs/`、`signal_outcomes.csv`、`shadow_log.csv` のような実データは Git に入れない。
- 本番へコードを反映するときは、Git 管理下のファイルだけを同期する。
- 本番ログを確認したいときは、必要なログだけを別で取得する。

補足:

- これは「開発コードの履歴管理」と「本番で増える実データ管理」を分ける運用です。
- 実データまで Git に入れると、差分が増えすぎて判断しにくくなり、誤って本番ログを上書きしやすくなります。

## 5. 判断ルール

### 5-1. すぐ不具合扱いしない条件

- Ver02.1 は「通知が少ない」だけでは不具合扱いしない。
- `suppress_reason_codes` と `confidence` の整合が取れていれば、まず観察継続にする。

### 5-2. 不具合とみなす条件

以下のどれかが出たら、運用確認ではなく不具合調査へ切り替える。

- `heartbeat.txt` が更新されない
- `last_result.json` が止まる
- `trades.csv` に新規行が増えない
- 通知時に `notify_reason_codes` が空
- `daily-sync` 後に `signal_outcomes.csv` / `user_reviews.csv` が更新されない
- `actual_move_driver` を入れても `logic_validated` が反映されない

### 5-3. 閾値調整のルール

- 閾値調整は単発の印象で行わない。
- 少なくとも以下のどちらかを見てから判断する。
  - 数件の通知レビュー
  - 一定期間の抑制ログ

## 10. ブランチ命名ルール（Stable / Snapshot / Baseline）

### 10-1. 目的

- 名前だけで「更新する枝か」「固定して比較する枝か」を判別できる状態を保つ。
- `freeze` 表記は段階的に廃止し、意味が伝わる `stable` / `snapshot` / `baseline` を優先する。

### 10-2. 用語定義

- `main`
  - 常に最新版の入口。最新運用で使える状態を維持する。
- `verXX.Y-vN`
  - 継続開発ライン。日々の改修はここへ積む。
- `snapshot/verXX.Y`
  - ある時点の安定状態を固定保存する枝。比較や差し戻しの基準点。
- `verXX.Y-stable`
  - 世代内の安定固定版。過去安定版を参照しやすくするために使う。
- `verXX-baseline`
  - 比較基準として維持する運用枝。原則として更新しない。

### 10-3. 現在の対応表

- 最新入口: `main`
- 現在の開発ライン: `ver02.1-v1`
- 現在の固定スナップショット: `snapshot/ver02.1`
- 比較基準: `ver01-baseline`
- 安定固定版: `ver02.0-stable`

補足:

- 旧 `codex/*` 系ブランチは整理済み。以後はこの命名ルールだけを使う。

### 10-4. 履歴の残し方（秘書メモ）

- `👩‍⚖️秘書.md` の `ブランチ履歴（時系列）` は、ブランチ名を先頭に見て判断できる形で維持する。
- 1 レコードは `日時` + `ブランチ名` + `1行説明` の 3 点セットで書く。
- 日時は `YYYY.MM.DD HH:MM JST` で統一する。
- 新しい変更を上に追加し、古いものを下へ送る（最新→過去の順）。

## 11. NEXT_TASK と progress の分離ルール

- `NEXT_TASK.md` は「次に判断するためのメモ」だけを書く。
- `progress.md` は「何を実施したかの履歴」を書く。
- `NEXT_TASK.md` に時系列の実施履歴を残さない。履歴化し始めたら削って `progress.md` へ移す。
- `NEXT_TASK.md` で残すのは、現在の前提、次アクション、ブロッカー、完了条件だけにする。
- 詳細な背景説明、比較ログ、長い補足は `progress.md` か `運用資料/reports/` へ分離する。
