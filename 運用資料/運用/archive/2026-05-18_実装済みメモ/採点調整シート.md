# 採点調整シート

判定が慎重すぎる、または攻めすぎると感じたときの早見表です。現行の主対象は `Ver02.4-v1` です。

今の主論点は、通知の有無そのものよりも、`どの価格で待つか / 入るか / 切るか / 利確するか` の精度です。  
まず `daily-sync` のレポートと AI 事後評価を見て、`SL` `TP` `15分足の執行価格` のどこが弱いかを先に切り分けます。

1. `運用資料/reports/feedback_daily_sync_YYYYMMDD.md`
2. `運用資料/計画/AI事後評価運用_Ver02.4-v1.md`
3. `運用資料/計画/Phase1条件の見方.md`
4. `logs/csv/user_reviews.csv`
5. `logs/signals/<signal_id>.json`
6. `.env`
7. `config.py`

## 今の見方

- `primary_setup_status=ready`
  - 条件がかなり揃った「検討可能」状態です
- `phase1_active=true`
  - `ready` の中でも、方向・価格・データ品質まで含めて Phase 1 の本番観測対象になった状態です
- `sl_eval`
  - `good / too_tight / too_loose`
- `tp_eval`
  - `good / too_close / too_far`
- `tf_15m_eval`
  - 実際に入る価格の精度を見る評価です。今はここを最重要で見ます

## 迷ったときの対応

- 無駄な通知が多い
  - `ATTENTION_ALERT_*`
  - `CONFIDENCE_*`
  - `src/notification/trigger.py`
- 通知方向は合うが、入る価格が微妙
  - `user_reviews.csv` の `tf_15m_eval`
  - `sl_eval`
  - `tp_eval`
  - `logs/review/chart_snapshots/`
- `ENTRY_OK` なのに最終的に待機や invalid になる
  - `logs/signals/<signal_id>.json` の `prelabel` と `primary_setup_status`
  - `src/analysis/position_risk.py`
  - `src/analysis/confidence.py`
- `ready` が全然出ない
  - `運用資料/計画/Phase1条件の見方.md`
  - `src/analysis/rr.py`
  - `src/trade/activation.py`
- AI 事後評価が増えない
  - `tools/log_feedback.py daily-sync`
  - `tools/log_feedback.py sync-ai-post-reviews`
  - `AI_POST_REVIEW_DAILY_MAX`

## 今の優先論点

- `15分足` の執行価格精度
  - 再検討帯が広すぎないか、狭すぎないか
- `SL` 設計
  - ノイズで刈られやすいか
- `TP` 設計
  - 近すぎるか、遠すぎるか
- `ready` / `phase1_active` の母数
  - レビュー件数より、本番で条件が立つかを優先して見る

## 原則

- 単発の印象で動かさない
- まず `daily-sync` のレポートと AI 事後評価の複数件を見る
- 実際に走る値は `config.py` より `.env` を優先して確認する
- 数値を動かす前に、`15分足評価` と `SL/TP評価` の偏りを確認する
