# ⭕️BTC判定システム 改善仕様書パックの使い方

## 概要
このパックは、`afrogdesign/BTC_FX_CODEX` の評価ロジック改善を、CODEX に **正確に引き継いで作業させる** ための最小かつ実務的なセットです。

今回の主眼は、**現行ロジックをいきなり壊さずに、Ver02.x の現在地から Ver03 / Ver04 / Ver05 へ繋がる改善仕様と作業導線を repo に正式保存すること**です。

---

## このパックの中身

### `00_readme.md`
このファイルです。  
役割、使う順番、注意点をまとめています。

### `01_prompt.md`
CODEX にそのまま貼るための **統合プロンプト** です。  
これが一番重要です。

### `02_master.md`
今回の改善仕様・保存先・追記案・実装タスク・コミット / PR 案を **1本にまとめたマスター文書** です。  
CODEX が repo に正式文書を配置する時の元資料になります。

---

## 使う順番

1. `01_prompt.md` を CODEX に貼る  
2. 必要なら `02_master.md` も同時に見せる  
3. CODEX に repo を読ませる  
4. repo 内の既存計画文書と整合を取りながら、正式文書を配置・更新させる

---

## 期待している作業
CODEX には、主に次をやらせます。

- repo の現状確認
- 既存のロードマップ / 計画書 / 運用資料の確認
- 改善仕様書の正式配置
- `NEXT_TASK.md` と `progress.md` の追記
- 次の実装に進むためのタスク整理
- 必要ならブランチ名 / コミット文 / PR 文面の提案

---

## 重要な前提
今回の目的は、**大規模なコード改修そのものではありません。**

優先順位は次の通りです。

1. 計画文書との整合確認
2. 改善仕様書の正式化
3. shadow 指標・観測基盤の仕様整理
4. 実装タスクの分解
5. その後にコード改修

---

## 絶対に外したくない方針

- 既存の最終判定をいきなり変えない
- 閾値チューニングを主目的にしない
- 通知件数だけを理由に調整しない
- 1回の変更で複数フェーズを飛ばさない
- まず観測可能性と比較可能性を上げる
- repo の計画文書を正本として扱う

---

## 一番大事なファイル
迷ったら、まず `01_prompt.md` を使ってください。  
その次に `02_master.md` を参照すれば足ります。
