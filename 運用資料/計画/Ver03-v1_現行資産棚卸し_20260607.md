# Ver03-v1 現行資産棚卸し

## 1. 結論

Ver02.6-v2 までの資産は、市場状況と活用実績から取捨選択が必要。

主要ロジックである `market_map`、サポート・レジスタンス、entry zone、SL/TP、paper_positions、report hub は維持する。

一方で、`trade_execution_gate=pass` だけを実質的な GO と扱う現行運用は、BTC の実務には厳しすぎる。

Ver03-v1 では、正式 gate を高信頼レーンとして残しつつ、毎回の通知で「成行」「指値戻り待ち」「ブレイク追随」「逆方向短期」「保有中の処理」を提示する `Active Trade Plan` レイヤーを新設する。

## 4. 廃止または降格する考え方

以下の考え方は廃止または降格する。

- 正式 `trade_execution_gate=pass` が出るまで、実務上の行動計画を出さない運用
- watch 通知を「実行不可」だけで終わらせる運用
- 大局方向を件名で強調し、実際に取れる行動が読みにくい通知
- `下方向バイアス` を `今ショートしてよい` と誤読させる構成
- `上方向バイアス` を `今ロングしてよい` と誤読させる構成
- 「危ないから入るな」だけの改善
- `trend_flip_confirmed_up` を強評価へ戻す方針

## 5. 現行 gate が実用トレードに向かない理由

現行 gate は安全性を優先する高信頼 gate としては有効。

しかし、BTC の実用トレード支援としては厳しすぎる。

理由:

- `primary_setup_status == ready` でないと `phase1_active` にならない。
- `watch` がほぼ実行対象外になる。
- `trade_execution_gate=pass` が長期間 0 件になっている。
- BTC では戻り待ち・押し目待ち・サポート反発・レジスタンス反落・ブレイク後の再テストが多い。
- その多くは `ready` ではなく `watch` や `invalid` 寄りに分類される。
- 現行通知は、方向と実行可否を分けているが、実務上の次アクションが弱い。
- 人間は方向スコアを見て入りやすい。
- しかし、システムは「どこなら入れるか」「今入るならRRはいくらか」「保有中ならどうするか」を十分に出していない。

## 6. BTC 実務に必要な新機能

Ver03-v1 で必要な新機能は以下。

- `Active Trade Plan` 生成
- 現在価格と entry zone の位置判定
- 現在価格で入った場合の RR 計算
- zone mid で入った場合の RR 計算
- 成行可否の判定
- 指値・戻り待ち候補の提示
- ブレイク追随候補の提示
- 逆方向短期スキャル候補の提示
- 保有中ポジションの利確・建値撤退・損切り目安
- 通知件名の Active Plan 優先化
- HTML hero の Active Plan 優先化
- Active Plan の日次診断
- Active Plan 別の紙検証
- 将来的な manual position tracking

## 7. Ver03-v1 の最初の実装候補

最初の実装候補は以下の順番。

1. `src/trade/active_plan.py` を追加する。
2. `tests/test_active_trade_plan.py` を追加する。
3. `20260606_230500` 型の fixture で、成行 short blocked、limit short allowed、counter long conditional をテストする。
4. `main.py` に `active_trade_plan` payload を接続する。
5. 通知件名と HTML hero を Active Plan 優先にする。
6. `tools/log_feedback.py` に `active_trade_plan_diagnostics` を追加する。
7. Active Plan の紙検証レーンを追加する。
