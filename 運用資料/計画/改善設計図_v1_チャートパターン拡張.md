# 改善設計図 v1 チャートパターン拡張

更新日: 2026-03-24 10:48 JST

この文書は、`Ver02.3` 時点の BTC Monitor に対して、
「人間がチャートを見るような時系列読解」をどこまで機械化するかを、
実装前に具体化するための設計図です。

ここでいう改善対象は、次の 6 つです。

1. 名前つきチャートパターンの検出
2. 短期窓と長期窓の分離
3. comparison の市場局面別比較
4. 出来高構造の強化
5. support / resistance 文脈の長期化
6. external data 欠損時の代替ロジック整備

この文書は「すぐ実装するコード仕様」ではなく、
段階採用のための正本設計として使います。

## 1. まず結論

- いきなり最終判定を置き換えない。
- 先に `shadow` レイヤーとしてパターン認識を追加する。
- 最初の導入対象は、`bull_flag`、`bear_flag`、`ascending_triangle`、`descending_triangle`、`failed_breakout` の 5 類型に絞る。
- comparison は全体集計だけでなく、`market_regime`、`phase`、`pattern_flags` 別に比較できるようにする。
- 本採用は、「全体成績が少し良い」だけではなく、「局面別に悪化していない」ことを条件にする。

## 2. 背景

現状のシステムは、以下の読解はできています。

- `hh_hl` / `lh_ll` の構造判定
- breakout / range center 判定
- EMA 配列と傾き
- support / resistance、流動性、板、清算、OI/CVD を使った位置評価

一方で、次はまだ未実装です。

- 上昇フラッグ
- 下降フラッグ
- ペナント
- 三角持ち合い
- 失敗ブレイク
- 形状と出来高を組み合わせた継続 / 反転パターン判定

そのため、現在は
「方向感と危険回避は比較的強いが、人間が一目で分かる形の良し悪しは弱い」
という状態です。

## 3. 目的

- 人間のチャート読解に近い「形の文脈」を、説明可能なルールとして追加する。
- 現在の `bias`、`prelabel`、`confidence` をすぐ変更せず、比較可能な `shadow` 指標として先に保存する。
- comparison を局面別・パターン別に見られるようにし、改善の当たり外れを可視化する。
- 将来の `Ver03.x` から `Ver05` に向けた分析基盤を、無理なくつながる形で用意する。

## 4. 非目的

- いきなり AI にチャート画像を読ませること
- OpenAI API を使った自由なパターン命名
- 通知件数を増やすためだけの閾値緩和
- 現在の最終判定を 1 回の変更で全面置換すること
- 本番の毎時運用へ未検証ロジックを即反映すること

補足:

- この設計では、まずルールベースの追加を優先します。
- ChatGPT API を使ったパターン認識は、別段階として扱います。

## 5. 基本方針

- 新ロジックは最初は `shadow` 保存のみ
- 既存 top-level 出力は維持
- comparison は「全体」だけでなく「条件別」に比較
- 1 回の変更で複数改善を本採用しない
- 良かった項目だけを段階的に `score / confidence / setup` へ取り込む

## 6. 現在ロジックの弱点整理

### 6.1 時系列窓が用途別に十分分かれていない

- 4h `300本`
- 1h `500本`
- 15m `500本`

は保持しているが、実際の判定は直近 `20本` 前後へ寄りやすい。

### 6.2 構造は見ているが、形状認識がない

- `hh_hl` は見ている
- `breakout_up` は見ている
- しかし「押しの角度」「保ち合いの収束」「出来高縮小から再拡大」は見ていない

### 6.3 comparison の粒度が粗い

- 現状は `baseline / rebalanced` の全体比較が中心
- `range` では改善、`transition` では悪化、のような局面差が見えにくい

### 6.4 external data 欠損時の落ち方が粗い

- 板や清算が欠けると中立寄りに落ちる
- 欠損時の代替的な価格構造判定が弱い

## 7. v1 の設計対象

v1 では、次の 5 レイヤーを追加または再設計する。

1. `pattern_context`
2. `window_context`
3. `volume_structure`
4. `sr_context`
5. `fallback_context`

## 8. 新設する概念

### 8.1 `pattern_context`

目的:

- 名前つきチャートパターンの検出結果を保存する

最初に扱う候補:

- `bull_flag`
- `bear_flag`
- `ascending_triangle`
- `descending_triangle`
- `failed_breakout`

初期保存項目案:

- `pattern_context.version`
- `pattern_context.detected_patterns`
- `pattern_context.primary_pattern`
- `pattern_context.pattern_scores.long`
- `pattern_context.pattern_scores.short`
- `pattern_context.pattern_reason_codes`
- `pattern_context.pattern_conflicts`

### 8.2 `window_context`

目的:

- 長期・中期・短期・トリガーの窓を明確に分ける

初期保存項目案:

- `window_context.macro_window`
- `window_context.structure_window`
- `window_context.trigger_window`
- `window_context.micro_window`

### 8.3 `volume_structure`

目的:

- 単発の `volume_ratio` だけでなく、保ち合い時の出来高縮小と breakout 時の拡大を見る

初期保存項目案:

- `volume_structure.contraction_score`
- `volume_structure.expansion_score`
- `volume_structure.breakout_volume_confirmed`
- `volume_structure.volume_regime`

### 8.4 `sr_context`

目的:

- 直近だけでなく、中長期の節目を別枠で持つ

初期保存項目案:

- `sr_context.short_horizon_levels`
- `sr_context.mid_horizon_levels`
- `sr_context.long_horizon_levels`
- `sr_context.nearest_major_resistance`
- `sr_context.nearest_major_support`

### 8.5 `fallback_context`

目的:

- external data 欠損時に、どの代替材料で判定したかを明示する

初期保存項目案:

- `fallback_context.mode`
- `fallback_context.missing_fields`
- `fallback_context.substituted_by`
- `fallback_context.confidence_penalty`

## 9. 履歴窓の再設計

### 9.1 基本思想

同じ 15 分足でも、目的別に見る長さを分ける。

### 9.2 v1 で固定したい窓

| 層 | 主時間足 | 初期窓 | 用途 |
| --- | --- | --- | --- |
| `macro_window` | 4h | 200-300 本 | 大局トレンド、EMA200、長期構造 |
| `structure_window` | 1h | 80-160 本 | 押し目 / 戻り売り、保ち合いの形の土台 |
| `trigger_window` | 15m | 20-40 本 | breakout、entry、直近勢い |
| `micro_window` | 15m | 60-120 本 | flag や triangle の形状判定 |

補足:

- v1 では fetch 本数の即変更を必須にしない。
- まずは既存取得範囲内で窓を分離し、必要なら後で `FETCH_LIMIT_15M` の増加を検討する。

## 10. パターン検出の具体案

### 10.1 `bull_flag`

定義イメージ:

- 先行上昇がある
- その後の押しが浅い
- 押しの角度は急落でなく、緩い下降または横ばい
- 保ち合い中に出来高が縮む
- 上抜け時に出来高が再拡大する

初期ルール案:

- 直近 `N1` 本で上昇幅が `k1 * ATR` 以上
- その後 `N2` 本の高値切り下げ・安値切り下げが緩い
- 押し幅が先行上昇幅の `0.25` から `0.55` に収まる
- 保ち合い中平均出来高 < 先行上昇中平均出来高
- breakout 時 `volume_ratio >= TRIGGER_VOLUME_RATIO`

### 10.2 `bear_flag`

`bull_flag` の逆。

### 10.3 `ascending_triangle`

定義イメージ:

- 上値抵抗がほぼ水平
- 安値が切り上がる
- 上抜け候補として見る

初期ルール案:

- 直近高値群のばらつきが ATR 比で小さい
- 直近安値群が段階的に上昇
- breakout 前に価格が収束している

### 10.4 `descending_triangle`

`ascending_triangle` の逆。

### 10.5 `failed_breakout`

定義イメージ:

- breakout はしたが、定着せずレンジ内へ戻る

初期ルール案:

- 直近 `20本` の高値 / 安値を明確に超える
- しかし `M本` 以内にレンジ内へ回帰
- 出来高拡大が弱い、または wick rejection が強い

## 11. 出来高構造の強化

現状:

- `volume_ratio` は直近出来高 ÷ 20 本平均

不足:

- 保ち合いで縮んでいるか
- breakout で再拡大したか
- volume の推移が価格パターンと整合しているか

v1 の追加案:

- `contraction_score`
  - 保ち合い区間の平均出来高が、その直前 impulsive leg より何割縮んだか
- `expansion_score`
  - breakout 足の出来高が保ち合い平均より何倍か
- `volume_regime`
  - `contracting` / `neutral` / `expanding`

## 12. support / resistance 文脈の長期化

現状:

- 各時間足の直近スイング最大 10 個中心

不足:

- 長期節目と短期節目が同じ箱に入っている
- 強い節目と直近ノイズが混ざりやすい

v1 の追加案:

- `short_horizon_levels`
  - 直近実務用の節目
- `mid_horizon_levels`
  - 1h 中心の節目
- `long_horizon_levels`
  - 4h 中心の主要節目
- `major_level_strength`
  - 長期側へ重みを寄せた強度

## 13. external data 欠損時の代替ロジック

### 13.1 基本方針

- 欠損したから即中立、ではなく
- 「どの情報が欠けたので、何で代替したか」を明示する

### 13.2 初期代替ルール案

| 欠損 | 代替 |
| --- | --- |
| orderbook 欠損 | wick rejection、近接ゾーン、直近高安密集度で代替 |
| liquidation 欠損 | swing cluster、直近高安集中帯で代替 |
| OI 欠損 | 価格変化 + 出来高 + CVD のみで弱い判断へ落とす |
| CVD 欠損 | OI と出来高のみで方向補助 |

### 13.3 出力

- `fallback_context.mode`
  - `normal`
  - `price_only_partial`
  - `structure_only`
- `fallback_context.confidence_penalty`

## 14. comparison の再設計

### 14.1 全体比較だけでは不足

必要な比較粒度:

- `market_regime` 別
- `phase` 別
- `pattern_flags` 別
- `fallback_mode` 別
- `data_quality_flag` 別

### 14.2 v1 の追加出力案

新規比較 CSV 候補:

- `comparison_by_regime.csv`
- `comparison_by_phase.csv`
- `comparison_by_pattern.csv`
- `comparison_by_fallback_mode.csv`

最低限の列:

- `label`
- `segment_key`
- `n`
- `ready_signals`
- `filled_trades`
- `avg_realized_rr`
- `profit_factor`
- `max_drawdown_rr`

## 15. shadow 導入の段階設計

### Step 0 設計固定

- この文書を正本として追加
- 既存計画書との整合を取る

### Step 1 観測追加のみ

- `pattern_context`
- `window_context`
- `volume_structure`
- `sr_context`
- `fallback_context`

を JSON に保存する

この段階では:

- `bias`
- `prelabel`
- `confidence`
- `signal_tier`

は変えない

### Step 2 shadow score 化

- `pattern_score_long_shadow`
- `pattern_score_short_shadow`
- `volume_quality_shadow`
- `sr_context_shadow`

を追加し、comparison 可能にする

### Step 3 segmented comparison

- regime / phase / pattern ごとの比較 CSV を出す

### Step 4 controlled adoption

- 効果が確認できた一部だけを `score / confidence / setup` に反映する

- まず JSON を正本にする

## 18. 採用条件

本採用の条件は、最低でも次を満たすこと。

1. 全体 comparison で悪化していない
2. `range`、`transition`、`uptrend`、`downtrend` の主要局面で致命的悪化がない
3. パターン該当時だけ見た比較で改善が説明できる
4. `fallback_mode` 時に極端な誤判定増加がない
5. 本番の最終通知挙動を急に荒らさない

## 19. テスト方針

### 19.1 単体テスト

- 人工 OHLCV で各パターンの検出確認
- 誤検出しやすいケースの否定テスト

### 19.2 回帰テスト

- shadow 導入段階で既存の最終判定が変わらないこと

### 19.3 比較テスト

- 全体比較
- regime 別比較
- pattern 別比較
- fallback 別比較

### 19.4 実運用テスト

- `Ver02.3-OBS` などの観測環境で JSON 保存だけ先に確認

## 20. リスク

- パターンの定義を増やしすぎると過学習しやすい
- 条件が複雑すぎると説明不能になる
- 直近相場に最適化しすぎると regime が変わったときに崩れる
- comparison 粒度を増やしすぎると、母数不足で判断がぶれる

## 21. v1 完了条件

### 設計完了

- パターン候補が固定されている
- shadow 導入順が決まっている
- 保存先と comparison 粒度が決まっている

### 実装着手可能

- どのモジュールを追加するか決まっている
- どの段階で既存判定を変えないか決まっている
- comparison の新しい切り口が決まっている

## 22. 今すぐの具体タスク

1. `patterns v1` で扱う 5 類型の判定ルールを、OHLCV ベースでさらに数式化する
2. `pattern_context` の JSON 形式を固定する
3. comparison のセグメント CSV 仕様を固定する
4. `shadow only` 実装の着手条件を `開発ロードマップ.md` と `NEXT_TASK.md` に反映する
5. 実装はその後に、`pattern_context` 保存から始める

## 23. 将来との接続

- `Ver02.3` では設計固定と shadow 観測
- `Ver03.x` では limited adoption
- `Ver04` では通知品質改善と組み合わせ
- `Ver05` では A/B 比較モードと内部スコア化へ接続

補足:

- v1 の狙いは「いきなり賢くする」ことではありません。
- まずは、チャートの形を比較可能な形で保存し、
  「効いたパターンだけを残す」ための土台を作ることです。
