# 改善設計図 v1 チャートパターン拡張

更新日: 2026-03-24 12:53 JST

この文書は、`Ver02.3` 系で扱うチャートパターン拡張の正式設計書です。
役割は「実装前の構想メモ」ではなく、実装者がそのまま着手できる `shadow-only` 仕様の正本です。

## 1. 結論

- v1 の範囲は `shadow-only` に固定する。
- 既存の `bias`、`prelabel`、`confidence`、`signal_tier`、通知判定、CSV ヘッダは v1 では変更しない。
- 新しい観測項目は result 直下へ増やさず、`chart_pattern_shadow` という単一 wrapper 配下へまとめる。
- `chart_pattern_shadow` は AI 判定や通知判定に渡す payload へ含めず、永続化直前にだけ merge する。
- live 保存先は `logs/last_result.json` と `logs/signals/*.json`、backtest 側は `run_backtest()` の返り値 dict のみとする。
- `comparison_by_regime.csv` などの segmented comparison と、`score / confidence / setup` への反映は v1 に含めず、後段へ送る。

## 2. この設計の位置づけ

この設計は、次の前提に従います。

- `Phase 0` 本番確認を最優先とする。
- `評価システム改善仕様書_Ver02x-Ver05接続.md` の「観測追加と判定変更を混ぜない」原則を守る。
- 人間のチャート読解に近い形状情報を入れたいが、v1 では説明用 shadow として保持するだけに留める。

したがって v1 の目的は、「最終判定を変えずに、チャートパターン文脈を保存できる状態を作ること」です。

## 3. v1 でやること

- `bull_flag`
- `bear_flag`
- `ascending_triangle`
- `descending_triangle`
- `failed_breakout`

上の 5 類型を、OHLCV と既存の swing / breakout / volume 情報からルールベースで検出する。

追加する観測領域は次の 5 つです。

1. `pattern_context`
2. `window_context`
3. `volume_structure`
4. `sr_context`
5. `fallback_context`

これらは単独の top-level key ではなく、`chart_pattern_shadow` の下へ保存します。

## 4. v1 でやらないこと

- `bias`、`prelabel`、`confidence`、`signal_tier` の変更
- 通知条件や通知件数を増やすための閾値緩和
- `trades.csv`、`shadow_log.csv`、`evaluation_*.csv`、comparison CSV の列追加
- `comparison_by_regime.csv`、`comparison_by_phase.csv`、`comparison_by_pattern.csv` の出力
- ChatGPT API や画像認識によるパターン命名
- `FETCH_LIMIT_*` の変更
- `score / confidence / setup` への直接反映

## 5. 公開インターフェース

### 5.1 保存する top-level key

v1 で新規追加する top-level key は次の 1 つだけです。

- `chart_pattern_shadow`

この key は次の JSON にのみ保存します。

- `logs/last_result.json`
- `logs/signals/*.json`

この key は v1 では次に保存しません。

- `last_notified.json`
- `last_attention_notified.json`
- CSV 系ファイル

### 5.2 `chart_pattern_shadow` の構造

```json
{
  "chart_pattern_shadow": {
    "version": "v1",
    "pattern_context": {},
    "window_context": {},
    "volume_structure": {},
    "sr_context": {},
    "fallback_context": {}
  }
}
```

### 5.3 `pattern_context`

- `detected_patterns`
  - 最大 5 件
  - 各要素は `{name, side, score, state, breakout_ready, reason_codes, reference_prices, window_label}`
- `primary_pattern`
- `pattern_scores`
  - `{long, short}`
- `pattern_conflicts`

補足:

- `pattern_scores` は shadow 値です。既存 score へ加算しません。
- `state` は `active` `forming` `invalidated` の 3 値で固定します。

### 5.4 `window_context`

- `macro_window`
  - `{timeframe:"4h", bars:240}`
- `structure_window`
  - `{timeframe:"1h", bars:120}`
- `trigger_window`
  - `{timeframe:"15m", bars:24}`
- `micro_window`
  - `{timeframe:"15m", bars:96}`

### 5.5 `volume_structure`

- `contraction_score`
- `expansion_score`
- `breakout_volume_confirmed`
- `volume_regime`
- `reference_leg_bars`
- `consolidation_bars`

`volume_regime` は次の 3 値で固定します。

- `contracting`
- `neutral`
- `expanding`

### 5.6 `sr_context`

- `short_horizon_levels`
- `mid_horizon_levels`
- `long_horizon_levels`
- `nearest_major_support`
- `nearest_major_resistance`
- `major_level_strength`

各 horizon の level 数は最大 5 件に制限します。
各 level は `{low, high, strength, timeframe, distance_from_price}` に固定します。

### 5.7 `fallback_context`

- `mode`
  - `normal`
  - `price_only_partial`
  - `structure_only`
- `missing_fields`
- `substituted_by`
- `confidence_penalty_shadow`

補足:

- `confidence_penalty_shadow` は参考値であり、v1 では `confidence` に適用しません。

## 6. 窓の固定値

v1 は既存 fetch 範囲の中で用途別の窓だけを分離します。

| key | timeframe | bars | 用途 |
| --- | --- | --- | --- |
| `macro_window` | 4h | 240 | 大局トレンド、長期節目 |
| `structure_window` | 1h | 120 | 押し目 / 戻り売り、形の土台 |
| `trigger_window` | 15m | 24 | breakout、直近 entry 条件 |
| `micro_window` | 15m | 96 | flag / triangle / failed breakout 判定 |

## 8. パターン判定ルール

### 8.1 `bull_flag`

次をすべて満たしたら候補とする。

- 直前 impulsive leg が `2.2 ATR` 以上
- 押し戻し比率が `0.25` から `0.55`
- 保ち合い本数が `6` から `20`
- 保ち合い中平均出来高が impulsive leg 中平均出来高の `0.85` 以下
- breakout 足で `volume_ratio >= 1.15`

### 8.2 `bear_flag`

`bull_flag` の逆条件で判定する。

### 8.3 `ascending_triangle`

次をすべて満たしたら候補とする。

- 直近 3 swing high のばらつきが `0.35 ATR` 以下
- 直近 3 swing low が段階的に切り上がる
- 各切り上がり幅が `0.15 ATR` 以上
- `micro_window` 内で収束が進んでいる
- breakout 足で既存 breakout / volume 条件を満たす

### 8.4 `descending_triangle`

`ascending_triangle` の逆条件で判定する。

### 8.5 `failed_breakout`

次をすべて満たしたら候補とする。

- 既存 breakout level を `0.15 ATR` 以上抜ける
- `4` 本以内にレンジ内へ回帰する
- wick rejection か volume 失速を伴う

## 9. module 責務

v1 の責務分割は次で固定します。

- `src/analysis/patterns.py`
  - 5 類型の判定
- `src/analysis/volume_structure.py`
  - contraction / expansion / volume_regime
- `src/analysis/sr_context.py`
  - horizon 別 level 抽出
- `src/analysis/fallback_context.py`
  - 欠損情報から fallback mode を構成

既存再利用は次で固定します。

- `structure.py`
  - swing 抽出の土台
- `breakout.py`
  - breakout 基準の土台
- support / resistance 構築処理
  - `sr_context` の入力元

## 10. main 側の統合順

`main.py` では次の順に扱います。

1. 既存の market / score / setup / notify 用 result を `core_result` として組む
2. `request_ai_advice()`、`should_notify()`、`build_summary_body()` は `core_result` だけで実行する
3. `chart_pattern_shadow` は別オブジェクトで組む
4. 永続化直前に `core_result` へ merge する
5. `logs/last_result.json` と `logs/signals/*.json` へ保存する

この順序にする理由は次です。

- AI payload へ shadow 情報を混ぜないため
- 通知挙動を v1 で変えないため
- result の top-level をむやみに増やさないため

## 11. backtest 側の扱い

`backtest/runner.py` では、各 signal dict に `chart_pattern_shadow` を持たせます。

ただし v1 では次を行いません。

- `evaluation_*.csv` への列追加
- comparison summary への列追加
- `tmp/backtest/*.json` の追加保存

v1 の backtest 目的は、「後段で比較に使える context を signal dict 上へ載せること」までです。

## 12. fallback 設計

`fallback_context` は `MarketStructureSnapshot.missing_fields` を起点に組み立てます。

初期ルールは次で固定します。

| 欠損 | 代替 |
| --- | --- |
| `orderbook_*` 欠損 | wick rejection、近接 zone、直近高安密集度 |
| `liquidation_events` 欠損 | swing cluster、直近高安集中帯 |
| `oi_*` 欠損 | 価格変化 + volume のみで弱い判断 |
| `cvd_series` 欠損 | OI と volume のみで方向補助 |

mode の初期ルールは次です。

- 欠損なし: `normal`
- external data の一部欠損だが価格構造と出来高で代替可能: `price_only_partial`
- external data ほぼ欠損で、構造中心の観測に落とす: `structure_only`

## 13. テスト方針

### 13.1 単体テスト

- 5 パターンの陽性ケース
- 誤検出しやすい否定ケース
- `fallback_context.mode` の分岐

### 13.2 回帰テスト

- 同一入力で `bias`
- `prelabel`
- `confidence`
- `signal_tier`
- `primary_setup_status`

が変わらないこと

### 13.3 保存テスト

- `logs/last_result.json`
- `logs/signals/*.json`

に `chart_pattern_shadow` が保存されること

### 13.4 副作用防止テスト

- AI へ渡す payload に `chart_pattern_shadow` が含まれないこと
- 通知判定に `chart_pattern_shadow` が影響しないこと
- CSV ヘッダが変わらないこと

## 14. 後段へ送る項目

次は v1 の外です。

- `comparison_by_regime.csv`
- `comparison_by_phase.csv`
- `comparison_by_pattern.csv`
- `comparison_by_fallback_mode.csv`
- `pattern_scores` の score / confidence / setup 反映
- `chart_pattern_shadow` の通知本文活用
- fetch 本数の見直し

## 15. リスク

- pattern 定義を増やしすぎると過学習になりやすい
- `chart_pattern_shadow` を AI payload に混ぜるとコストと挙動がぶれる
- config 定数を追加しても型変換を忘れると文字列のまま動く
- level 数を増やしすぎると JSON が肥大化する
