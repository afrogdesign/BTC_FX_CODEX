# BTC Monitor マイルストーン定義

更新日: 2026-03-12 11:10 JST

このファイルは、`Ver02` 以降の世代番号を「何ができたら上げるか」で固定するための定義書です。
日々の小さな改善と、大型節目での世代交代を分けて扱います。

## 基本ルール

- 日々の小さな改善は `Ver02.x`、`Ver03.x` のような小版で扱う
- 大型節目を満たしたときだけ、次の世代番号へ上げる
- 世代番号を上げるときは、同じ安定コミットに対して以下を同時に行う
  - 新ブランチ作成
  - 世代タグ付与
- 例:
  - `Ver03` 到達時: `ver03` ブランチ + `v3.0` タグ
  - `Ver04` 到達時: `ver04` ブランチ + `v4.0` タグ

## 現在地

- 現在の開発ブランチ: `ver02`
- 既存タグ: `v1.0`, `v2.1`, `v2.2`, `v2.3`
- 現在の世代位置づけ: `Ver02.x`

## Ver02.x

- 定義:
  - `Phase 0` 実運用確認と `Phase 1` 着手前後の改善版
- 主な作業:
  - 通知後フロー一周確認
  - `Phase 1` モジュールの設計と導入
- 小版タグの使い方:
  - ログ強化
  - ドキュメント整備
  - 軽微な調整

## Ver03

### 昇格条件

- `Phase 0` の通知後フローが本番で一周完了している
- `was_notified=True` を本番データで確認済み
- `notify_reason_codes` が本番ログへ保存されることを確認済み
- `daily-sync` を本番で 1 回以上完走済み
- `actual_move_driver` 入力後の `logic_validated` 反映を確認済み
- `Phase 1` の中核である `position_sizing` と `exit_manager` が導入済み
- サイズ計画と出口計画が JSON / CSV に保存されている

### 意味

- `Ver03` は「観測システム」から「損益管理つき監視システム」へ進んだ節目

### Git 運用

- 安定コミットで `ver03` ブランチを作成する
- 同じコミットへ `v3.0` タグを付ける
- その後の軽微修正は `Ver03.x` / `v3.x` として扱う

## Ver04

### 昇格条件

- `Phase 2` の主要機能が live データで稼働している
- blackout が動作している
- 価格乖離監視が動作している
- `watch_prices` が表示・保存されている
- AI審査が構造化出力で記録されている
- `config_change_log` が運用開始されている
- 通知監査 A/B/C/D が安定運用できている

### 意味

- `Ver04` は「通知品質と安全性」を備えた世代

### Git 運用

- 安定コミットで `ver04` ブランチを作成する
- 同じコミットへ `v4.0` タグを付ける

## Ver05

### 昇格条件

- `Phase 3` の主要基盤が稼働している
- 因子相関分析ツールが使える
- prelabel 内部スコア化が導入されている
- A/B比較モードが運用できる
- `build_setup()` 精密化が比較可能な形で進められる
- シナリオ分岐出力が導入されている

### 意味

- `Ver05` は「分析結果を根拠にロジック改善できる」世代

### Git 運用

- 安定コミットで `ver05` ブランチを作成する
- 同じコミットへ `v5.0` タグを付ける

## 昇格判断時の確認項目

- 本番で何を実測確認したか
- 何を実装済みにしたか
- 何をまだ未解決として残すか
- `progress.md`、`NEXT_TASK.md`、`👩‍⚖️秘書.md`、`📒打ち合わせノート.md` をどこまで更新したか

補足:

- 世代番号は見た目のかっこよさではなく、大型節目の整理に使う
- 小版タグは「安定版のしおり」、世代番号は「段階が変わった節目」として分けて扱う
