# Ver03-v1 統合再計画

更新日: 2026-06-08 JST
対象repo: `afrogdesign/BTC_FX_CODEX`
対象branch: `Ver03-v1`

## 1. 結論

Ver03-v1 は、単に通知を増やす改修ではなく、BTC の実務に合わせて「人間が使える行動計画」を生成し、それを将来の自動売買へ接続できる形で検証する再設計フェーズである。

現時点の最上位方針は以下。

- まずメール通知で人間が取引する。
- 将来的には自動トレードシステムへ昇華する。
- ただし、実弾発注 API、注文送信、秘密鍵連携はまだ行わない。
- `trade_execution_gate` は壊さず、高信頼 gate として残す。
- `FORMAL_GO` と `ACTIVE_*` を混同しない。
- Active Plan は、実弾発注ではなく実務行動計画である。
- BTC では成行追随より、戻り待ち、押し目、再テスト、サポレジ反応、保有中処理が重要。
- 次の本筋は、Active Plan 候補が本当に機能したかを intraperiod で検証することである。

## 2. Ver03-v1 の目的

Ver03-v1 の目的は、毎回のメール通知を以下の状態へ近づけること。

```text
今は入るべきか。
入らないなら、どこまで待つべきか。
入るなら、SL / TP / RR はどうなるか。
保有中なら、利確・建値撤退・損切りのどこを見るべきか。
それが将来自動化できるほど検証可能な形で記録されているか。
```

## 3. 残す資産

以下は Ver03-v1 以降も土台として残す。

- `market_map`
- support / resistance 自動検出
- support-resistance flip / failed breakout 系
- entry zone
- SL / TP / RR
- `paper_positions.csv` の `pending -> opened -> closed`
- `missed_opportunity`
- `entry_not_reached`
- `tp1 / tp2 / sl / timeout` 後追い評価
- report hub
- daily-sync
- `tools/log_feedback.py` の既存 builder / CLI 群
- 既存 CSV ログ
- `trade_execution_gate`

ただし、`trade_execution_gate` は「実用トレード判断の唯一の GO」ではなく、高信頼レーンとして残す。

## 4. 降格・廃止する考え方

以下は降格する。

- `trade_execution_gate=pass` が出るまで、実務上の行動計画を出さない運用
- watch 通知を「実行不可」だけで終わらせる運用
- 方向スコアを件名で強調し、実際に取れる行動が読みにくい通知
- `下方向バイアス` を `今ショートしてよい` と誤読させる構成
- `上方向バイアス` を `今ロングしてよい` と誤読させる構成
- `ACTIVE_*` を正式 GO や自動発注候補として扱うこと

## 5. Active Plan の位置づけ

Active Plan は、正式 gate を緩める機能ではない。

分類は以下。

```text
FORMAL_GO
ACTIVE_MARKET_SMALL
ACTIVE_LIMIT_RETEST
ACTIVE_BREAKOUT_FOLLOW
ACTIVE_COUNTER_SCALP
POSITION_MANAGEMENT
NO_ACTION
```

扱い:

- `FORMAL_GO`: 既存 gate 通過。将来自動化候補。
- `ACTIVE_MARKET_SMALL`: 成行小ロット候補。最も慎重に扱う。
- `ACTIVE_LIMIT_RETEST`: Ver03-v1 の主戦場。戻り待ち・押し目・再テスト。
- `ACTIVE_BREAKOUT_FOLLOW`: 出来高・確定足・再テスト条件つき。
- `ACTIVE_COUNTER_SCALP`: 主に利確・建値撤退警告。新規は conditional 基本。
- `POSITION_MANAGEMENT`: 保有中の利確・撤退・損切り目安。
- `NO_ACTION`: 期待値がない局面を除外する重要分類。

Active Plan の第一段階は、おおむね実装済みとして扱う。

接続済みの範囲:

- Active Plan core
- main payload
- 通知件名
- HTML hero
- `trades.csv` logging
- diagnostics
- effectiveness
- paper candidates
- candidate outcomes
- candidate outcomes report
- daily-sync 接続
- report family registry

ただし、本格的な紙検証はまだ未完成である。

現在の candidate outcomes は forward close ベースの暫定評価であり、以下をまだ見ていない。

- entry zone に届いたか
- entry 後に TP1 / TP2 / SL のどれへ先に触れたか
- timeout になったか
- candidate entry price 基準の MFE / MAE

## 7. 次にやるべきこと

次にやるべきことは、`active_plan_candidate_intraperiod_outcomes` の設計と最小実装である。

ただし、いきなり daily-sync や既存 `paper_positions.csv` に混ぜない。

順序:

1. 計画フォルダの正本を再構成する。
2. `active_plan_candidate_intraperiod_outcomes` の実装方針を確定する。
3. OHLCV 入力の扱いを決める。
4. 単体 builder を実装する。
5. fixture で long / short / not_entered / ambiguous / timeout / pending を固定する。
6. report builder を作る。
7. 十分に安定してから daily-sync に接続する。
8. さらに安定してから、既存 `paper_positions.csv` との関係を再検討する。

## 8. 実装優先順位

### P0: 計画と正本の整理

目的:

- どの計画書を読むべきか迷わない状態にする。
- 古い `ver02.6-v2` 前提を正本から外す。
- Ver03-v1 の現在地を明確にする。

成果物:

- `README.md` 更新
- 統合再計画
- 計画フォルダ再構成案
- AI参照ファイル設計

### P1: intraperiod 検証の最小実装

目的:

- Active Plan 候補が、実際に entry できたかを検証する。
- entry 後に TP / SL のどちらが先だったかを見る。
- candidate entry price 基準で MFE / MAE を計算する。

成果物:

- `active_plan_candidate_intraperiod_outcomes.csv`
- `build_active_plan_candidate_intraperiod_outcomes()`
- 単体テスト

### P2: intraperiod report

目的:

- action / candidate_type / side / entry_mode ごとの実用性を見る。
- `ACTIVE_LIMIT_RETEST` が主戦場として機能しているか確認する。
- `ACTIVE_MARKET_SMALL` が過剰売買になっていないか確認する。
- `ACTIVE_COUNTER_SCALP` が conditional として妥当か確認する。

### P3: メール通知の実用化

目的:

- メールだけで人間が次の行動を判断できる状態にする。
- 成行 / 指値 / ブレイク / 逆方向 / 保有処理を、件名と本文で誤読なく出す。

### P4: 自動化直前部品

目的:

- kill switch
- blackout
- size cap
- daily loss limit
- manual position tracking
- 自動発注前の安全装置

ここまで完了しても、ユーザーが明示許可するまで実弾発注は行わない。

## 9. 禁止事項

- 実弾発注 API を追加しない。
- 取引所 API キーや秘密鍵を扱わない。
- `ACTIVE_*` を正式 GO と表示しない。
- `trade_execution_gate=pass` と `ACTIVE_*` を混同しない。
- intraperiod 検証が安定するまで、既存 `paper_positions.csv` に Active Plan 候補を混ぜない。
- OHLCV がない場合に無理な外部取得をしない。
