BTC Monitor 計画全体タイムライン

このページは、運用資料/計画/ に分かれている複数の計画書を、 人間が一目で理解できるように時系列へ並べ直した総覧です。 左の太い縦線を「大きな歴史の流れ」とし、その節目ごとに 「何が完了したか」「いま何をしているか」「次に何をするか」「将来どうつながるか」を 見える化しています。

このページでは、英語の変数名や内部用語が出てきたら、できるだけその場で 「日本語でいうと何か」「その数字や状態が出ると何が分かるか」を添えています。 つまり、コードを知らなくても「今は前に進める段階か」を判断できるようにした版です。

先に読んでおく用語の意味

用語 日本語でいうと これが分かると何を判断できるか
daily-sync 毎日の状況をまとめる日次レポート処理 今日の運用状況を、細かいログを見なくても先頭だけで把握できます。
ready 次の段階に進めそうな条件がそろった状態 Phase 1 に進む候補が本番で出始めたかを確認できます。
phase1_active=true Phase 1 の判断が実際に有効になった状態 「候補が出た」だけでなく、「本当に次段階が動いた」かを確認できます。
primary_setup_status その通知候補がどの段階まで整っていたかを示す状態名 ready が増えると、次段階の候補が増えていると判断できます。
ai_audit AI がその通知をどう見たか残す監査メモ 最終的な通知文や順位づけが直感に合っているか確認できます。
AI事後評価 通知を出した数時間後から 24 時間後までの結果を、AI が後追いで採点する仕組み 「方向が合っていたか」だけでなく、「どの価格で入るべきだったか、SL と TP は妥当だったか」を振り返れます。
4時間足 / 1時間足 / 15分足 大きな流れ、帯の妥当性、実際の執行価格を分けて見る3段チャート 上段で方向、中段で再検討帯、下段で入る価格・SL・TP を確認できます。いちばん重要なのは下段の 15 分足です。
shadow 本番の判定にはまだ使わず、裏で記録だけする試験観測 既存の判定を壊さずに、新しい見方が有効かを比べられます。
was_notified=True その通知が実際に送られた状態 机上ではなく、本番でユーザーへ届いたケースかを見分けられます。
position_sizing どれくらいの大きさで入るか決める仕組み 通知だけでなく、実運用のリスク管理が入っているか確認できます。
exit_manager どこで利確・撤退するかを管理する仕組み 入口だけでなく出口まで運用設計が入ったか確認できます。
watch_prices 人が次に見るべき価格帯を示す監視情報 将来、通知の後に何を注視するべきかを案内できるようになります。

大きな歴史の流れ

完了

土台形成期: Ver02.x の観測基盤

過去から現在へつながる基礎

まず最初に整えたのは、「通知が良かったか悪かったかを後から追える土台」です。 ここで観測基盤とログ基盤が先にできたことで、今後の改善が比較可能になりました。

大きな流れ
  • shadow_log.csv: 裏で試験記録をためる観測台帳
  • signal_outcomes.csv / user_reviews.csv: 通知結果と人の感想を残す台帳
  • notify_reason_codes / suppress_reason_codes: 通知した理由、しなかった理由の記録
  • data_quality_flag: その時のデータが信用できるかの印
小さな流れ
  • signal_based / entry_ready_based: 通知直後だけでなく、後から結果を振り返る基盤
  • 週次・月次レポート基盤: 日々の結果を後からまとめて比較できる
  • Phase 1 計画値保存列: 将来必要になる数字を先に保存しておく土台
意味

まだ賢くする前に、「何が効いて、何が悪かったか」を記録できる状態を作った段階です。

現在

Phase 0 本番観測と Phase 1 前倒し確認

いまいちばん実務に近い流れ

いまの中心は、Ver02.5-v7 を本番観測しながら、 通知後 24 時間評価、Phase 1A、Phase 1B-lite、 そして 15 分足の執行価格精度を同時に見ることです。

現在できていること
  • Ver02.5-v7 を主系統として運用中: いま見るべき中心版はこれ
  • 通知時だけ ai_audit を保存する形へ切替済み: 必要な場面だけ AI 監査メモを残す運用に整理済み
  • 通知時の詳細 HTML は 3 段チャート化済み: 上段 4 時間足、中段 1 時間足、下段 15 分足で価格帯・SL・TP を確認できる
  • 通知後は AI 事後評価が自動で走る: 通知で示した価格帯・SL・TP が実務上妥当だったかを後から採点できる
  • daily-sync に Phase 1 判定サマリー と判定文を追加済み: 今日進めてよいかを日次レポートで即確認できる
  • 15分足 執行チェック を追加: 主要ライン直近の追いかけ ready を watch に落とせる
  • レビューは JSON 正本 + HTML フォームで維持できる: 人の感想も残せるが、主役は AI 事後評価に切り替わった
いま詰めていること
  • primary_setup_status=ready が live で何件出るか: 危ない ready が減り、実務に近い ready が残るかを見る
  • Phase1B-lite が 10-15 件に届くか: 現在 5 件のため、正式昇格ではなく観測継続
  • 実メールで最終ランクと 3 段チャートが直感に合うかの確認: 人が読んで「どこで入るべきか」が分かる通知になっているか確認する
  • SL評価 / TP評価 / 15分足評価 の偏り確認: 価格精度のどこを優先改善するかを絞る
  • 2026-05-18 時点: daily 47 件、概算 PF 0.73、勝率 46.8%、formal pass 0 件
この流れの完了条件
  • Phase 0 の通知後フローを本番で 1 周完了: 通知して終わりではなく、後追い評価まで一巡させる
  • レビューは最低 5 件あれば補助観測として十分にする: 人の感想は残すが、それ以上は前進の足止めにしない
  • Phase 1 正式指標を実データで確認可能にする: 実際の数字で次段階を判定できる状態にする

設計固定

内部観測強化の枝: evaluation_trace / shadow 観測

本流を壊さずに比較材料を増やす流れ

この枝は、今の最終判定を変えずに「なぜそう判定したか」を後から比較できるようにするものです。 すぐに賢くするのではなく、先に説明可能な内部観測を増やす位置づけです。

設計済み
  • evaluation_trace.direction_score_shadow: 方向感が合っていたかを裏で採点する記録
  • entry_quality_score_shadow: 入る場所の良さを裏で点数化する記録
  • trigger_quality_score_shadow: 通知を出したきっかけの良さを裏で点数化する記録
  • risk_component_scores: 危険度を分解して見るための部品点数
まだ未着手
  • 通知停止権限のような強い介入はまだやらない
  • 既存判定を壊さないまま補助観測だけ増やす
  • 実データ確認後にだけ次段階を再検討する
将来の役割

Ver04 / Ver05 で、A/B 比較や内部スコア化へつながる橋渡しになります。

新設計

将来分析の枝: チャートパターン拡張 v1

人間のチャート読みへ近づけるための未来枝

これは今すぐ着手中の本線ではありません。上昇フラッグや三角持ち合いのような 名前つきパターンを、将来の分析材料として shadow 観測へつなぐための設計枝です。

最初に扱う類型
  • bull_flag: 上昇途中の一時休みのような形
  • bear_flag: 下落途中の一時戻しのような形
  • ascending_triangle: 上に抜けやすい三角持ち合い
  • descending_triangle: 下に抜けやすい三角持ち合い
  • failed_breakout: 抜けたように見えて失敗した形
将来やる順番
  • Step 0: 設計固定。何を見たいか先に決める
  • Step 1: pattern_context などを JSON 保存。形の情報を後から読めるよう残す
  • Step 2: shadow score 化。本番判定に使わず裏で点数化する
  • Step 3: segmented comparison。パターン別に効果を比較する
  • Step 4: 効いたものだけ controlled adoption。本当に効いたものだけ慎重に採用する
この枝の意味

「人間のようなチャート読み」をいきなり本番へ入れず、 比較可能な設計枝として先に増やす流れです。

本流との関係

この枝は現在の実務線を止めてまで進めるものではありません。 本流を置き換えず、将来の Phase 3 / Ver05 側の分析枝へ接続します。

次の大型節目

Ver03: 観測システムから損益管理つき監視システムへ

Ver02.x の次に超えるべき線

Ver03 は、見た目の番号上げではなく、 Phase 0 一周完了 + Phase 1 中核導入確認 を満たした時だけ名乗る節目です。

必要条件
  • was_notified=True の本番確認: 通知が机上ではなく本当に送られていること
  • daily-sync 本番完走: 日次まとめが毎日安定して出せること
  • phase1_active=true と ready の本番記録確認: 次段階の候補と実発動の両方が確認できること
  • サイズ計画 / 出口計画の本番ログ確認: どれくらい入るか、どこで終えるかも本番ログに残ること
すでにある土台
  • position_sizing: どれくらいの大きさで入るか決める仕組み
  • exit_manager: 利確や撤退の管理をする仕組み
  • phase1_active / phase1_activation_reason: Phase 1 が動いたか、その理由は何かを残す列
  • risk_percent_applied などの保存列: どれくらいのリスク設定を使ったか後から確認できる
意味

Ver03 は「ただ通知する」から「損益管理を伴う監視」へ進んだことを示す節目です。 ただし、今はまだその手前で、Phase 1 の本有効確認を前倒ししている段階です。

未来

Phase 2: 通知品質と危険回避の強化

Ver04 側へつながる安全装置の流れ

Phase 2 は、通知の質と「危ない時に止まる力」を増やす層です。 人が次に見る価格や、異常時の警戒を返せるようにします。

大きな流れ
  • watch_prices: 次に人が見るべき価格帯を示す
  • 価格乖離監視: 参照価格とずれすぎた時に気づけるようにする
  • blackout: 危険な時間帯やイベント前後に通知を止める仕組み
  • 構造化 AI 審査出力: AI の判断理由を決まった形で残す
  • config_change_log: 設定変更の履歴を残す台帳
小さな流れ
  • 通知前の危険警告
  • 人向け再評価価格の提示
  • イベント前後の停止
  • AI 自由文依存の削減
到達先

Phase 2 が live で回ると、Ver04 の「通知品質と安全性を備えた世代」へつながります。

未来

Phase 3: 分析して改善できる世代へ

Ver05 側へつながる分析枝

Phase 3 は「感覚で改善する」段階を終わらせ、 比較ログを根拠にロジックを改善するための分析層です。

分析基盤
  • 因子相関分析
  • A/B 比較モード
  • prelabel 内部スコア化
ロジック改善
  • build_setup() 精密化: 通知の組み立てロジックをより現実に合わせて細かくする
  • シナリオ分岐出力: どういう場合分けで見ているかを出せるようにする
  • 比較根拠つきの変更運用: 勘ではなく比較結果を根拠に改善する
チャート拡張との接点

今回のチャートパターン拡張 v1 は、この Phase 3 の比較・分析枝と強く接続します。
