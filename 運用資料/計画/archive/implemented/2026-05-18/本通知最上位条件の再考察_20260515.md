# 本通知最上位条件の再考察

更新日: 2026-05-15 JST

## まず結論

- 現行の最上位 `strong_main` は、BTC Monitor の実ログではまだ一度も出ていない。
- これは単なるバグというより、BTC の値動きの性質と現行条件の組み合わせ上、かなり自然な結果に見える。
- `strong_main` は「本当に実行候補として条件が整った状態」に近く、現在の運用で多く発生している `watch` や `Phase 1A` 観測候補とは役割が違う。
- したがって、`strong_main` を出すために実行条件を緩めるのは危険。必要なら、実行候補の最上位とは別に「監視候補の上位」を作るほうがよい。

## 現行仕様の整理

現在の通知は、次の 3 層が混ざって見えやすい。

1. 送るかどうか
   - `main`
   - `attention`
   - `none`
2. 表示ランク
   - `strong_main`
   - `high_main`
   - `normal_main`
   - `attention`
   - `no_send`
3. 実行候補かどうか
   - `trade_execution_gate=pass`
   - `trade_execution_gate=blocked`

このうち、ユーザーが言う「本通知の最上位」は表示ランク上の `strong_main` に相当する。
ただし `strong_main` が出ても、それだけで実行してよいという意味ではなく、最終的には `trade_execution_gate=pass` と `paper_orders planned` が必要になる。

## `strong_main` の条件

`strong_main` は `signal_tier=strong_machine` のときだけ出る。

`strong_machine` には、少なくとも次が必要になる。

- `prelabel=ENTRY_OK`
- `primary_setup_status=ready`
- confidence が方向別の下限より 10 以上高い
- `rr_estimate >= 1.8`
- `no_trade_flags` が空
- major warning が 0 件
- minor warning が 1 件以下
- `agreement_with_machine=agree`

さらに表示側では、`confidence_execution_shadow <= 20` かつ `confidence_wait_shadow >= 60` の場合は、条件が強くても `normal_main` へ落とす。
つまり `strong_main` は「方向が強い」だけでは足りず、価格位置、引き金、RR、警戒要因、執行しやすさまで同時に通る必要がある。

## 実ログで起きていること

ローカル実績を見ると、最上位はまだ空席に近い。

- `logs/signals/*.json` 1491 件
  - `strong_main=0`
  - `high_main=98`
  - `normal_main=22`
  - `attention=72`
- `logs/csv/shadow_log.csv` 1454 行
  - `signal_tier=strong_machine`: 0 件
  - `primary_setup_status=ready`: 2 件
  - `ENTRY_OK + ready`: 2 件
  - `trade_execution_gate=pass`: 1 件

一方で `high_main` は出ている。
つまり「通知が弱すぎる」のではなく、「実行候補として完全に整った最上位だけが出ていない」という状態。

直近の `feedback_daily_sync_20260514.md` でも、Phase 1 は次の状態だった。

- `primary_setup_status=ready`: 0 件
- `phase1_active=true`: 0 件
- `trade_execution_gate=pass`: 0 件
- `paper_orders planned`: 0 件

このため、現時点では `Phase 1B` の厳格紙トレードへ進めない判断は妥当。

## BTC の性質から見た妥当性

BTC は、株式のように取引時間が区切られた市場とは違い、常に価格形成が続く。
CME の Bitcoin Real Time Index は、1秒ごと、24時間、365日で価格を出す前提の指数になっている。

また、SEC の Investor.gov は bitcoin / ether を高い投機性とボラティリティを持つ資産として説明している。
FINRA も、暗号資産は伝統的な投資資産より高いボラティリティを経験しており、流動性の薄さが価格変動を増幅しうると説明している。

この前提に立つと、BTC では次が同時に起きやすい。

- 方向は強いが、執行位置はまだ悪い
- レジサポが近く、RR が伸びにくい
- スイープ待ち、板の偏り、清算クラスターが残りやすい
- 15分足の引き金が出る頃には、すでに短期ノイズが増えている
- 上位足の方向と短期の執行条件がずれやすい

つまり BTC では「方向優位は見えるが、実行条件はまだ整わない」という状態が多くなりやすい。
現行ログで `watch`、`SWEEP_WAIT`、`RISKY_ENTRY` が多く、`ready` がほとんど出ないのは、この市場特性と矛盾しない。

## 判断

現行の `strong_main` は不可能な条件ではない。
ただし、今の BTC Monitor の観測粒度と BTC の相場特性では、かなり希少な状態として設計されている。

そのため、次のように整理するのがよい。

- `strong_main`
  - 厳格な実行候補に近い最上位として維持する。
  - 頻出しない前提でよい。
- `high_main`
  - 方向や監視価値が強いが、まだ実行候補ではない通知として読む。
  - 現在の運用では実質的な上位通知の中心。
- `normal_main`
  - 本通知だが、実行不可・監視継続・注意点優先の通知。
- `attention`
  - 方向変化や初動を早めに共有する通知。

問題は `strong_main` が出ないこと自体ではなく、`high_main` が「実行候補に近い」と誤読されること。
この誤読を避けるには、実行候補の最上位と監視候補の上位を分けて扱うほうがよい。

## 次に検討する設計案

### 案A: `strong_main` は据え置く

`strong_main` は「極めて厳格な実行候補寄りの最上位」として残す。
条件は緩めない。

利点:

- 安全側の設計を維持できる。
- `trade_execution_gate=pass` との整合を壊しにくい。

欠点:

- 実運用上はほぼ見ないランクになりやすい。
- 「最上位が出ないなら壊れているのでは」という疑問は残る。

### 案B: 監視候補の上位ランクを別に作る

たとえば `market_structure_high_watch` のように、実行候補ではないが監視価値が高い状態を別ランクにする。

候補条件は、次のような方向が合う。

- `phase1_observation_gate=pass`
- `market_map` の主要 flag が出ている
- `direction` は強い
- `trade_execution_gate=blocked`
- 件名と本文で「実行候補ではない」を明示する

利点:

- BTC で多い「方向は強いが執行不可」の状態を正しく上位扱いできる。
- `strong_main` を無理に緩めずに済む。

欠点:

- ランク体系が少し増える。
- 文面を間違えると、また実行推奨に見えるリスクがある。

### 案C: `strong_main` を非表示または廃止する

現実の運用ランクを `注意報 / 通常の本通知 / 高め本通知` に絞る。

利点:

- 表示体系がシンプルになる。
- 空席の最上位に悩まなくてよい。

欠点:

- 将来 `trade_execution_gate=pass` が自然に出始めたとき、区別する表示を再設計する必要がある。

## 推奨

現時点の推奨は、案Aと案Bの組み合わせ。

- `strong_main` は緩めず、厳格な実行候補寄りの最上位として残す。
- ただし運用上は、当面 `strong_main` が出ない前提でよいと明記する。
- 別途、Phase 1A / `market_map` 用の「監視候補上位」を検討する。
- `high_main` や監視上位の本文には、`trade_execution_gate=blocked` の場合に「実行候補ではない」を必ず出す。

この方針なら、BTC の高ボラ・連続市場・スイープ待ちの多さを受け入れつつ、実行ゲートを不用意に緩めずに済む。
