# 通知ランク再設計 実装設計

更新日: 2026-05-15 JST

## 目的

`strong_main` を無理に出しやすくするのではなく、通知の表示上で次を明確に分離する。

- 執行候補
- 高優先監視
- 通常監視
- 注意報
- 送信なし

BTC では方向や構造が強くても、位置、RR、スイープ待ち、待機圧力のために `trade_execution_gate=blocked` になりやすい。
その状態を `high_main` と表示すると実行候補に見えるため、`high_watch` を新設する。

## 最終ランク

| final_rank_code | 表示名 | 意味 |
|---|---|---|
| `strong_main` | 執行候補・強 | 実行候補として条件がかなり整っている |
| `high_main` | 執行候補 | 実行ゲートを通過した執行候補 |
| `high_watch` | 高優先監視・実行不可 | 方向・構造は強いが、実行候補ではない |
| `normal_main` | 通常監視・実行不可 | 監視と再評価のための本通知 |
| `attention` | 注意報・売買非推奨 | 方向変化や初動の共有 |
| `no_send` | 送信なし | メール送信条件は未成立 |

## 判定順

`src/presentation/sanitize.py` の `_final_rank()` は次の順で判定する。

1. `notification_kind=attention` は `attention`
2. `notification_kind=main` 以外は `no_send`
3. `trade_execution_gate=pass` かつ `paper_order_status=planned`
   - `signal_tier=strong_machine` なら `strong_main`
   - それ以外は `high_main`
4. `phase1_observation_gate=pass` は `high_watch`
5. それ以外の `main` は `normal_main`

`trade_execution_gate` が欠けている旧データは `blocked` とみなし、執行候補ランクには上げない。

## 件名と本文

件名は `final_rank_label` をそのまま使う。
そのため、メール一覧だけで `執行候補`、`高優先監視・実行不可`、`通常監視・実行不可`、`注意報・売買非推奨` を判別できる。

本文冒頭には必ず次を表示する。

- `trade_execution_gate`
- `trade_execution_blockers`
- `phase1_observation_gate`
- `phase1_observation_type`
- `phase1_observation_reasons`
- `paper_order_status`

`attention` は「これは売買推奨メールではありません。」を冒頭固定にする。

## テスト

最低限、次を固定する。

- 実行ゲート通過 + `paper_order_status=planned` + `strong_machine` は `strong_main`
- 実行ゲート通過 + `paper_order_status=planned` + `normal` は `high_main`
- 実行ゲート blocked + 観測ゲート pass は `high_watch`
- 実行ゲート blocked + 観測ゲート blocked は `normal_main`
- 注意報は `attention`
- 送信なしは `no_send`
- blocked の強い数値が `strong_main` / `high_main` にならない
