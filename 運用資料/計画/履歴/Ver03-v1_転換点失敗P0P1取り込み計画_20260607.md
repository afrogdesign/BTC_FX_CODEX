# Ver03-v1 転換点失敗 P0/P1 取り込み計画

作成日: 2026-06-07 JST  
対象ブランチ: Ver03-v1

## 1. 結論

Ver03-v1 は、Active Trade Plan の第一段階として、通知・HTML・CSV・diagnostics・effectiveness・candidate CSV・candidate outcomes・intraperiod outcomes builder まで進んだ。

ただし、外部提案書「BTC Monitor 転換点失敗の精査メモ・作業指示書」の指摘により、ここから先は Active Plan intraperiod report / daily-sync 接続へ直進しない。

次に優先するのは、既存の paper position / opportunity gate / trigger / paper diagnostics の土台修正である。

理由:

- `counter_long_short_watch` が observation paper order では反対 side を見る一方、paper position では primary setup の side / entry / stop / TP を保存している可能性がある。
- `missed_opportunity` と `entry_not_reached` が `realized_r` として扱われ、filled-only の実約定成績と混ざる可能性がある。
- `formal_execution_candidate` が hard quality conflict を blocker として扱っていない。
- 出来高急増だけで long / short 両方の trigger が立つ可能性がある。
- `trend_flip_confirmed_up` は strong long に戻すのではなく、まず short invalidation / short_reversal_risk として使うべきである。

したがって、次の大方針は以下。

1. Active Plan intraperiod outcomes builder は完了済みとして保持する。
2. ただし、intraperiod report / daily-sync 接続は一時停止する。
3. 先に P0 修正で paper position と成績集計の信頼性を直す。
4. 次に P1 修正で hard quality blocker、方向付き volume trigger、short_reversal_risk を入れる。
5. P0/P1 後の filled-only 成績を見てから、転換専用 entry と score / gate 緩和を検討する。

## 2. 取り込む提案書の要点

外部提案書の最重要結論は、現在の問題は単なる「安全すぎるスコア」ではなく、転換点を検知する材料が増えている一方で、最終的な entry / paper position / gate / レポート評価の流れが転換トレード向けに整っていないことだ、という点である。

提案書の優先順位:

1. 反転候補の side / entry / SL / TP 保存の不整合を直す。
2. `missed_opportunity` と `entry_not_reached` を実約定損益から分離する。
3. 出来高 trigger を方向付きにする。
4. `formal_execution_candidate` の hard quality conflict を blocker 化する。
5. 転換専用 entry / follow 条件を設計する。
6. 最後に score / gate の緩和を検討する。

この順序は Ver03-v1 でも採用する。

ただし、提案書の対象ブランチは `ver02.6-v1` とされていたため、Ver03-v1 では現在の実装状態に合わせて作業番号と対象ファイルを再整理する。

## 4. P0 修正計画

## P0-1. `counter_long_short_watch` paper position side / entry / SL / TP 修正

### 目的

`counter_long_short_watch` が paper position で primary setup の side / entry / SL / TP として保存される可能性をなくす。

### 現在確認できている事実

- `append_observation_paper_order()` は `phase1_observation_type == "counter_long_short_watch"` の場合、`bias=long` なら observation side を `short` に、`bias=short` なら `long` に反転している。
- `append_paper_position()` は `opportunity_type` を保存しているが、`side`、`entry_price`、`stop_loss_price`、`tp1_price`、`tp2_price`、`rr_estimate` は primary setup 系を使っている。

### 修正方針

`append_paper_position()` に helper を追加し、`opportunity_type == "counter_long_short_watch"` の場合だけ反対側 setup を使う。

- `bias == "long"` なら `short_setup`
- `bias == "short"` なら `long_setup`

使う値:

- `side`
- `entry_price`
- `stop_loss_price`
- `tp1_price`
- `tp2_price`
- `rr_estimate`

通常 opportunity は従来通り primary setup を使う。

### 実装予定作業

`BTCFX-20260607-055`

作業名:

`counter_long_short_watch paper position side 修正`

変更候補:

- `運用資料/作業ログ/Ver03-v1_counter_watch_paper_position_20260607.md`

検証:

- counter watch で side が反対側 setup になること。
- entry / SL / TP / RR が反対側 setup 由来になること。
- 通常 opportunity は従来通り primary setup になること。
- 既存 CSV header は変更しない。
- 全体 unittest。

## P0-2. filled-only / missed / entry_not_reached 集計分離

### 目的

`missed_opportunity` と `entry_not_reached` を実約定損益と混ぜない。

### 現在確認できている事実

- `paper_position.py` は pending 中に TP 方向へ先に走った場合、`missed_opportunity` として `_close_position()` し、`close_price=tp1` を使う。
- `paper_position.py` は entry が 24h 到達しなかった場合、`entry_not_reached` として `_close_position()` し、その時点の candle close を使う。
- `_close_position()` は `realized_r` を計算する。
- つまり、約定していないものも `realized_r` を持ちうる。

### 修正方針

互換性のため、既存 `realized_r` はすぐ削除しない。

ただし、report / diagnostics の主指標は以下に分ける。

- `filled_realized_r`
  - `opened` 後に `sl_hit` / `tp2_hit` / `timeout` で closed したものだけ
- `missed_r`
  - `missed_opportunity` 用
- `entry_not_reached_r`
  - `entry_not_reached` 用
- `all_paper_r`
  - 既存互換・参考用

最小実装では、まず report builder 側に filled-only 集計を追加する。  
CSV header 変更は必要最小限にする。

### 実装予定作業

作業番号:

`BTCFX-20260607-056`

作業名:

`paper diagnostics filled-only 集計分離`

変更候補:

- `src/trade/paper_position.py`
- `tools/log_feedback.py`
- diagnostics 関連テスト
- `運用資料/作業ログ/Ver03-v1_filled_only_paper_diagnostics_20260607.md`

検証:

- `missed_opportunity` が filled-only PF / 平均R / 勝率に入らないこと。
- `entry_not_reached` が filled-only PF / 平均R / 勝率に入らないこと。
- opened 後の `sl_hit` / `tp2_hit` / `timeout` は filled-only に入ること。
- 既存 `realized_r` 互換を壊さないこと。
- 全体 unittest。

## 6. P1 修正計画

P1 は P0 のあとに分割実装する。

## P1-1. formal candidate hard quality blocker 化

### 目的

`trade_execution_gate=pass` の formal candidate でも、hard quality conflict がある場合は pass させない。

### 現在確認できている事実

- `opportunity_gate.py` では `hard_quality_reason` があっても formal candidate の場合は `formal_candidate_quality_conflict:*` として reasons に追加される。
- blocker にはならない。

### 修正方針

`require_execution_for_high_wait` を含む hard quality reason は formal candidate でも blocker にする。

- hard quality reason あり:
  - `opportunity_gate=blocked`
  - `opportunity_type=blocked`
- soft quality reason のみ:
  - formal candidate は維持
  - reason に conflict を残す

作業番号候補:

`BTCFX-20260607-058`

## P1-2. 出来高 trigger 方向付き化

### 目的

出来高急増だけで long / short 両方の trigger が立つ構造を直す。

### 現在確認できている事実

`main.py` では現在、以下に相当する処理がある。

- `trigger_up = breakout_up or volume_ratio >= cfg.TRIGGER_VOLUME_RATIO`
- `trigger_down = breakout_down or volume_ratio >= cfg.TRIGGER_VOLUME_RATIO`

### 修正方針

15m candle の open / close / high / low と market_map flags を使って、volume trigger を方向付きにする。

long trigger:

- `breakout_up == true`
- または volume ratio 条件 + 陽線 + close が range 上側 / resistance 側
- または `failed_breakout_up_reversal`
- または `resistance_to_support_flip`

short trigger:

- `breakout_down == true`
- または volume ratio 条件 + 陰線 + close が range 下側 / support 側
- または `failed_breakout_down_reversal`
- または `support_to_resistance_flip`

doji に近い candle は出来高だけでは両方向 trigger にしない。

作業番号候補:

`BTCFX-20260607-059`

## P1-3. short_reversal_risk 追加

### 目的

上方向転換材料を、いきなり long 強化に戻さず、まず short invalidation / short suppression として使う。

### 修正方針

`long_reversal_risk` と対になる `short_reversal_risk` を追加する。

対象候補:

- `resistance_to_support_flip`
- `resistance_to_support_retest_confirmed`
- `failed_breakout_up_reversal`
- `trend_flip_confirmed_up`
- `trend_flip_early_up`
- `major_support_rejection`

confidence 側では、short bias 時に `short_reversal_risk` がある場合、execution を下げ、wait を上げる。

作業番号候補:

`BTCFX-20260607-060`

## 11. 最終判断

Ver03-v1 は、Active Plan を追加して「正式GOが少なすぎる」問題への対応を進めてきた。

しかし、ここで提案書の P0/P1 を先に取り込む。

理由は、勝てるシステムへ近づけるには、行動計画の表示だけでなく、以下の評価土台が正しくなければならないためである。

- 反転候補が正しい side で保存されていること。
- 約定していない仮想Rが実損益に混ざらないこと。
- hard quality conflict が formal candidate の抜け道にならないこと。
- 出来高 trigger が方向を持つこと。
- 上方向転換は、いきなりロング強化ではなく、まずショート無効化として働くこと。

この計画により、Ver03-v1 は Active Plan の実用化と、既存 paper / gate / trigger の信頼性修正を両立する。
