# Ver03-v1 AI参照ファイル設計

更新日: 2026-06-08 JST

## 1. 目的

ChatGPT、Codex、将来の自動化エージェントが、毎回どのファイルを読めばよいかを固定する。

## 2. ChatGPT が読む順番

### 2.1 通常の次作業判断

1. `運用資料/NEXT_TASK.md`
2. `運用資料/計画/00_Ver03-v1_統合再計画_20260608.md`
3. `運用資料/計画/02_Ver03-v1_実装ロードマップ_20260608.md`
4. `運用資料/reports/report_hub_latest.md`
5. 最新の Active Plan diagnostics / outcomes report

### 2.2 設計変更時

1. `00_Ver03-v1_統合再計画_20260608.md`
2. `正本/Ver03-v1_現行資産棚卸し_20260607.md`
3. `正本/Ver03-v1_Active_Trade_Plan_再精査_20260607.md`
4. `正本/Ver03-v1_Active_Plan_実装棚卸し_20260607.md`
5. `正本/Ver03-v1_Active_Plan_candidate_intraperiod検証設計_20260607.md`
6. `正本/Ver03-v1_Active_Plan_intraperiod実装前repo調査_20260607.md`

### 2.3 実装指示作成時

1. `NEXT_TASK.md`
2. 直近作業ログ
3. 対象の計画書
4. 対象コード
5. 対象テスト

## 3. Codex が読む順番

Codex は設計判断をしない。

Codex が読むのは以下に限定する。

1. ChatGPT が出した作業指示
2. その作業指示で指定された計画書
3. 対象コード
4. 対象テスト
5. `NEXT_TASK.md` の現在地

## 4. 将来の自動化エージェントが読むファイル

将来の自動売買化では、AIが読むファイルを減らす必要がある。

候補:

```text
ai_context/
  system_policy.md
  trading_policy.md
  risk_policy.md
  active_plan_policy.md
  manual_position_policy.md
  execution_forbidden_until_enabled.md
```

### 4.1 `system_policy.md`

- 実弾発注の許可状態
- 自動発注禁止フラグ
- 監視モード / paper mode / live mode

### 4.2 `trading_policy.md`

- `FORMAL_GO`
- `ACTIVE_*`
- `NO_ACTION`
- entry / TP / SL / RR の扱い

### 4.3 `risk_policy.md`

- 最大損失
- size cap
- daily loss limit
- kill switch
- blackout

### 4.4 `active_plan_policy.md`

- Active Plan action の意味
- human-only と auto-eligible の区別
- `ACTIVE_LIMIT_RETEST` の主戦場扱い
- `ACTIVE_MARKET_SMALL` の慎重扱い

### 4.5 `manual_position_policy.md`

- 手動ポジション入力仕様
- 保有中の利確・撤退・損切り表示ルール

### 4.6 `execution_forbidden_until_enabled.md`

- 取引所送信禁止
- APIキー / 秘密鍵を扱わない
- ユーザー明示許可まで live order 無効

## 5. レポート導線

report hub は残す。

ただし、Active Plan 系は以下の順番で見る。

1. `active_trade_plan_diagnostics_YYYYMMDD.md`
2. `active_trade_plan_effectiveness_YYYYMMDD.md`
3. `active_plan_candidate_outcomes_YYYYMMDD.md`
4. `active_plan_candidate_intraperiod_outcomes_YYYYMMDD.md`（今後）

## 6. 作業ログの扱い

作業ログは履歴であり、正本ではない。

ただし、直近作業の確認には使う。

毎回 hash placeholder 補正だけの作業を行う必要はない。今後は、重要な節目だけで作業ログを作る。

## 7. Codexプロンプト圧縮方針

同じCodexスレッドで継続する場合は、以下は圧縮する。

- 長い背景説明
- 何度も同じ禁止事項
- 既知の未追跡ファイル説明
- 完了報告フォーマットの過剰な反復

ただし、以下は必ず残す。

- 作業番号
- repo / branch
- HEAD
- 目的
- 変更対象
- 検証
- commit / push
- 今回重要な禁止事項

## 8. 結論

AI が読む入口を固定しない限り、古い計画や作業ログに引っ張られる。

Ver03-v1 では、このファイルと `00_Ver03-v1_統合再計画_20260608.md` を起点に、読み順を固定する。
