# Ver03-v1 実装ロードマップ

更新日: 2026-06-08 JST

## 1. 基本方針

Ver03-v1 は、メール通知で人間が取引し、将来は自動化へ進めるための実用トレード支援システムである。

したがって、優先順位は以下。

```text
1. 人間が誤読しない通知
2. 実際の entry / TP / SL 検証
3. 勝てる型と危険な型の分離
4. 紙検証の自動化
5. 自動化直前の安全装置
6. 実弾発注は最後
```

## 2. 現在完了済みとして扱うもの

- Active Plan core
- main payload 接続
- 通知件名 Active Plan 優先
- HTML hero Active Plan 優先
- `trades.csv` Active Plan logging
- Active Plan diagnostics
- Active Plan effectiveness
- Active Plan paper candidates
- Active Plan candidate outcomes
- Active Plan candidate outcomes report
- daily-sync 接続
- report family registry

## 3. まだ不十分なもの

- candidate outcomes は forward close ベースで、途中の高値・安値を見ていない。
- entry zone に実際に届いたかを見ていない。
- TP / SL のどちらが先だったかを見ていない。
- candidate entry price 基準の MFE / MAE を再計算していない。
- Active Plan 候補を既存 `paper_positions.csv` へ接続していない。
- 実ポジション管理はまだ未実装。
- 実弾発注 API は未実装のまま維持する。

## 5. P1: intraperiod outcome 最小実装

### 目的

Active Plan 候補が、実際に使える候補だったかを 15m 足で検証する。

### 入力

- `logs/csv/active_plan_paper_candidates.csv`
- 15m OHLCV

### 出力

- `logs/csv/active_plan_candidate_intraperiod_outcomes.csv`

### 判定

- entry reached
- entry reached time
- TP1 reached
- TP2 reached
- SL reached
- first exit
- timeout
- MFE
- MAE
- pending
- no_ohlcv

### 実装順

1. helper 関数のみ実装。
2. fixture test を追加。
3. builder を追加。
4. CLI を追加。
5. 実データで空でも落ちないようにする。
6. daily-sync にはまだ接続しない。

## 7. P3: 通知改善

### 目的

メールだけを見て、人間が取引判断できる状態にする。

### 改善項目

- 件名に Active Plan action を明確に出す。
- 本文冒頭に「今すぐ」「待ち」「見送り」「保有中なら」を出す。
- `NO_ACTION` は敗北ではなく、期待値なしとして表示する。
- `ACTIVE_LIMIT_RETEST` では、待つ価格帯と無効条件を出す。
- `ACTIVE_COUNTER_SCALP` は新規売買より、利確・撤退警告として優先表示する。
