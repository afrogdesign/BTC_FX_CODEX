このシステムは、どうやって育っていくのか

BTC Monitor には、ただメールを出すだけではなく、「その判断が本当に役に立ったのか」を後から見直す仕組みがあります。 この資料は、そのレビューの流れ、なぜ精度向上につながるのか、最終的に何を目指しているのかを、できるだけ分かりやすくまとめたものです。

結論を先に言うと

このシステムは「通知を出して終わり」ではなく、出した通知をあとで採点し、その採点をもとにロジックを直していく 仕組みです。 ただし、いまは完全自動の自己調整ではなく、機械が記録し、人が見て、必要な調整だけを入れる半自動方式 です。

まず、この仕組みは何のためにあるのか

  • 通知が多いか少ないかではなく、「役に立ったか」を見るためです。
  • あとから振り返った時に、「あの通知は早すぎた」「この見送りは正しかった」を残すためです。
  • 思いつきではなく、実際の記録をもとに閾値や判定を直すためです。

この仕組みが防ぎたいこと

  • 印象だけで「最近通知が少ないから緩めよう」としてしまうこと
  • 勝った通知だけ覚えて、負けた通知を忘れてしまうこと
  • 人によって「良かった / 悪かった」の基準が毎回ブレること

レビュー改善ループの全体像

1. 判定して通知する 毎サイクルで BTC の状態を見て、方向、信頼度、setup、位置リスクを決めます。必要ならメールを送ります。
2. その時の理由を保存する ただ通知するだけでなく、なぜ通知したか、なぜ通知しなかったか、何が危険だったかも保存します。
3. 24時間後に答え合わせする そのあと実際にどう動いたかを見て、「方向は合っていたか」「入る価値はあったか」を自動で後追い評価します。
4. 人がレビューを足す 役に立ったか、早すぎたか、遅すぎたか、自分なら入ったかなどを人が記録できます。
5. ロジックを調整する たまった記録を見て、「どこが過剰に厳しいか」「どこが甘いか」を判断し、必要な部分だけ直します。

何が自動で記録されているのか

記録されるもの 意味 何に使うか
last_result.json 最後の判定結果の生データ その回で何を判断したかを見る入口です。
shadow_log.csv 判定時のスコア、confidence、prelabel、signal_tier、通知理由などの一覧 傾向を見るための主力ログです。
signal_outcomes.csv 24時間後の自動事後評価 方向や timing が実際にどうだったかを見ます。
user_reviews.csv 人の感想や採点 機械だけでは分からない「実務で役立ったか」を残します。
notify_reason_codes / suppress_reason_codes 通知した理由 / 通知しなかった理由 「なぜメールが来たのか」「なぜ来なかったのか」を説明できます。

24時間後の自動評価とは何か

これは、通知のあとに市場がどう動いたかを後追いで採点する仕組みです。

  • 方向は合っていたか
  • entry 候補として意味があったか
  • 待機判断は正しかったか
  • 見送り判断は正しかったか

つまり、「その場ではもっともらしく見えた通知」が、あとで見ても本当に意味があったかを確認しています。

人のレビューとは何か

自動評価だけでは分からないことを、人が補います。たとえば次のような視点です。

  • 方向は合っていたが、通知が早すぎた
  • 実際には見送りで助かった
  • 本文は分かりやすかったが、件名が強すぎた
  • 自分なら条件付きで入った

この層があることで、「値動きは正しいが、実務判断では使いにくい」ケースも拾えます。

なぜこの仕組みで、取引の精度が上がるのか

仕組み 精度向上につながる理由
通知時の理由を保存する あとで「良かった通知」と「悪かった通知」を見比べた時に、共通する条件を見つけやすくなります。
24時間後に結果を採点する その場の印象ではなく、実際の相場結果で良し悪しを見られます。
人のレビューを残す 機械では測れない「使いやすさ」「早すぎる / 遅すぎる」を補正できます。
比較運転をする Ver02.1 と Ver02.3 を同時に見ることで、何が変わったかを感覚ではなく差分で確認できます。
単発ではなく複数件で見る たまたまの当たり外れではなく、傾向として「厳しすぎる / 甘すぎる」を判断できます。
重要なのは、「この仕組みがあるから必ず勝てる」という意味ではないことです。
正確には、どこが悪かったかを言語化しやすくなるので、改善の質が上がる、というのが根拠です。

この仕組みで実際に得られるもの

  • 通知の数ではなく、通知の質が見える
  • 「どの条件が過剰に厳しいか」が分かる
  • 「どの条件が甘すぎるか」が分かる
  • 本文や件名の分かりやすさも改善できる
  • 感覚ではなく、記録ベースで会話できる

この仕組みだけでは得られないもの

  • 完全自動で最適値に育つ保証
  • どんな相場でも勝てる万能ロジック
  • 少数サンプルだけで確実な結論

だから今は、完全自動ではなく「記録を増やし、根拠を持って直す」やり方を取っています。

いまの段階で、どこまで自動で回っているか

項目 今の状態 説明
判定結果の保存 自動 毎サイクル自動で残ります。
通知理由の保存 自動 なぜ通知したか / しなかったかが残ります。
24時間後の事後評価 半自動 daily-sync を回して同期します。
人のレビュー記入 手動 役に立ったかは人が入れます。
閾値の自動変更 未採用 いまは自動で勝手に緩めたり厳しくしたりしません。

どう観察すると、このレビュー仕組みが活きるのか

日常ではこの順で十分です

  1. 届いたメールを見て、「この通知は役に立ちそうか」を感覚で持つ
  2. 必要な通知だけ、あとで last_result.json や本文を見て理由を確認する
  3. 24時間後に signal_outcomes.csv で答え合わせする
  4. 必要なら user_reviews.csv に「早い / 遅い / 役に立った」を残す
  5. 数件たまってから、初めて閾値調整を検討する

特に見たい比較ポイント

  • Ver02.1 は watch だったが Ver02.3 は ready になったか
  • Ver02.3 の方が早いが、早すぎていないか
  • Ver02.3 が ready を増やしても、位置が悪い場面は prelabel で止めているか
  • メールの見た目と実際の値動きが一致しているか

この改善を続けると、最終的に何を目指しているのか

目指している完成形は、「通知が多いシステム」ではありません。次の3つが揃った状態です。

完成形の姿 意味
方向の質が安定する long / short / wait の出し方が、相場に対して過剰でも鈍すぎでもなくなることです。
入る場所の質が安定する 方向は合っていても、悪い場所なら止める設計が育つことです。
人が使いやすい通知になる 本文、件名、強さの出し方が、実務判断に使いやすくなることです。
もっと平たく言うと、完成形は
「いい場面では早めに気づけて、悪い場面では無理に入らず、あとで振り返っても納得できる通知システム」
です。

今後の自然な進み方

  • しばらく Ver02.1 と Ver02.3-OBS を並走観測する
  • 通知例を数件ためる
  • 24時間後評価を回す
  • 「どこが良くなったか」「どこがまだズレているか」を整理する
  • 必要な部分だけ次の Ver で直す

いまの段階で言えること

  • レビューの土台はすでにあります
  • 自動保存と事後評価の仕組みもあります
  • Ver02.3 で比較観測しやすい状態まで来ています
  • ここからは「作る段階」より「観察して育てる段階」です
