  • テーマ: 人間のチャート読みができているか

今の評価システムは、
「人間のようにチャートを読むAI」ではありません

ただし、まったく何も見ていないわけでもありません。現在の BTC Monitor は、 ローソク足の並びから 高値安値の更新、ブレイクアウト、 レンジ中央かどうか、EMA の向き、 上位足と下位足の整合 をルールベースで読んでいます。 その一方で、人間が言う「上昇フラッグ」「ペナント」「三角持ち合い」などを 名前つきで認識しているわけではありません。

最初に結論

結論として、今のシステムは「チャート画像を見てパターン認識するもの」ではなく、 時系列データに対して決め打ちの構造ルールを当てて読む仕組みです。 そのため、人間が当たり前に言う「上昇フラッグっぽいから上抜けを狙う」のような読みを、 そのままの形で理解しているわけではありません。

できていること

HH/HL、LH/LL、ブレイク、レンジ、転換帯の判定。

できていないこと

上昇フラッグ、ペナント、三尊などの名前つき形状認識。

今の性格

ルールベースの時系列判定システムです。

懸念への答え

「部分的には読んでいるが、人間並みのチャート読解ではない」です。

今のシステムが実際に見ているもの

見る対象 中身 人間の言い方に近い表現
スイング高値・安値 直近の高値安値の更新関係を見て hh_hl / lh_ll / mixed を作る 高値切り上げ・安値切り上げかどうか
EMA 配列と傾き 20 / 50 / 200 EMA の並びと向き 上位足が上昇トレンド寄りか
ブレイク判定 直近レンジ高値・安値を超えたか レンジ上抜け・下抜けか
レンジ中央判定 いまの価格が直近レンジの真ん中に近いか 中途半端な場所で入っていないか
複数時間足の整合 4h / 1h / 15m の向きがそろっているか 上位足と下位足がケンカしていないか

コード上の根拠

  • detect_swings() は高値安値の山谷を抜き出しています。
  • classify_structure() はそこから hh_hl と lh_ll を判定しています。
  • classify_market_regime() は上昇トレンド、下降トレンド、転換帯、レンジを分けています。
  • compute_scores() は breakout、range_center、transition_up_hint などを点数化しています。
  • つまり、完全なブラックボックスではなく、かなり明示的なルールの組み合わせです。
人間でいう「チャートの雰囲気を見る」を、今のシステムでは 高値安値・EMA・RSI・出来高・レンジ位置 に分解して読んでいる、と考えると近いです。

どこまで「人間のチャート読み」に近いか

1. 並びは見ている

ローソク足 1 本だけではなく、過去の並びから山谷と流れを見ています。

2. 方向感も見ている

EMA とスイング構造の整合で、上昇・下降・転換を判定します。

3. 位置も見ている

サポート、レジスタンス、レンジ中央、流動性の近さを見ています。

4. ただし形の名前は持たない

「これは上昇フラッグです」とラベルづけしているわけではありません。

5. だから一部は拾い、一部は逃す

上昇フラッグ的な場面でも、ルールに落ちない形は読み取れません。

人間なら言うこと 今のシステム 状態

ペナントだから次の上抜けを狙う 収束パターンを図形として認識していない 未実装

できている読み

  • 高値安値の切り上げ
  • 高値安値の切り下げ
  • 直近レンジの上抜け下抜け
  • 上位足と下位足の整合
  • サポレジ近辺の危険度

できていない読み

  • 上昇フラッグ
  • 下降フラッグ
  • ペナント

  • ダブルトップ / 三尊

いまの本質

  • 画像認識ではない
  • 名前つき図形認識ではない
  • 複数指標の合成判定
  • かなり説明可能
  • 人の経験則を一部分解した形

上昇フラッグ型のような「当たり前の予測」は、今どこまでできるか

厳密には、上昇フラッグそのものを理解しているわけではありません。 ただし、そのパターンを構成する一部の材料は見ています。

上昇フラッグを人間が見る時の要素 今のシステムで見ているか 補足
上位足が上昇トレンド 見ている EMA と hh_hl で近い判定をしています。
いったん小さく押している 弱く見ている 構造やレンジ扱いには出るが、「旗の傾き」としては見ていません。
持ち合い上限のブレイク 見ている 直近高値更新の breakout 判定があります。
出来高増加を伴う 見ている volume_ratio を使っています。
旗型という形そのもの 見ていない これが今の大きな欠落です。
つまり、「上昇フラッグに近い場面を、別の部品で部分的に拾うことはある」が、 「これは上昇フラッグだから上抜け期待」とまではまだ言えていません。

比較 CSV や事後評価は、この懸念を解決しているのか

いいえ、直接は解決していません。比較 CSV や事後評価は、 今あるルールが良いか悪いかを後から確かめるものです。

  • 比較 CSV: baseline と rebalanced を同じ過去データで比べる
  • 事後評価: 通知後 1h / 4h / 12h / 24h の結果を測る
  • どちらも「パターン認識能力そのもの」を新しく作っているわけではない

なので、「人間のようにフラッグや持ち合いを読めるか」という懸念に対しては、 比較 CSV の存在だけでは足りず、パターン検出の実装そのものが必要です。

実務的な答え

今の Ver02.3 は、チャートの流れをルールで読むシステムであって、 人間トレーダーが「これ旗だね」と一目で言う形を、そのまま理解するシステムではありません。 したがって、「当たり前のチャートパターン予測ができているか」という問いへの正直な答えは まだ部分的にしかできていないです。

もし本当にそこを強くしたいなら

  • 上昇フラッグ、下降フラッグ、ペナント、ダブルトップなどの専用検出ルールを追加する
  • 各パターンに「信頼度」「否定条件」「想定ターゲット」を持たせる
  • 今の score / confidence に、パターン認識の結果を別レイヤーで足す
  • comparison CSV でも「パターンあり / なし」で比較できるようにする
つまり次の進化は「今のルールを捨てる」ではなく、 今ある構造判定の上に、名前つきチャートパターン認識を追加するのが自然です。
